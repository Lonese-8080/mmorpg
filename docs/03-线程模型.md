## 1. 设计背景

### 1.1 为什么游戏服务器需要特殊的线程模型？

```
问题：通用服务器（Web API） vs 游戏服务器

Web API 的情况：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  请求 A ──▶ [处理1] ──▶ [数据库] ──▶ [返回]                │
│  请求 B ──▶ [处理2] ──▶ [数据库] ──▶ [返回]                │
│  请求 C ──▶ [处理3] ──▶ [数据库] ──▶ [返回]                │
│                                                              │
│  特点：                                                       │
│  - 每个请求是独立的                                           │
│  - 请求之间没有共享状态                                       │
│  - 可以随便并发、随便加锁                                      │
│                                                              │
└─────────────────────────────────────────────────────────────┘

游戏服务器的情况：
┌─────────────────────────────────────────────────────────────┐
│                                                              │
│  玩家A ──▶ [移动] ──▶ 与玩家B的坐标比较 ──▶ 战斗判断          │
│  玩家B ──▶ [移动] ──▶                                        │
│  玩家C ──▶ [释放技能] ──▶ 影响周围的玩家 ──▶ 计算伤害       │
│                                                              │
│  特点：                                                       │
│  - 所有玩家的状态都在同一个世界                               │
│  - 动作可能互相影响                                          │
│  - 需要严格的执行顺序                                         │
│  - 卡顿会影响玩家体验                                         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心矛盾

```
┌─────────────────────────────────────────────────────────────┐
│                      核心矛盾                                │
│                                                              │
│   ┌───────────────────┐      ┌───────────────────┐        │
│   │     正确性         │      │      性能          │        │
│   │                   │      │                   │        │
│   │ 多个线程同时修改   │      │   多核 CPU 充分利用  │        │
│   │ 同一份数据会导致   │      │   并行处理逻辑      │        │
│   │ 数据不一致         │      │                    │        │
│   └───────────────────┘      └───────────────────┘        │
│                                                              │
│   ❌ 方案1：加锁保护                                         │
│      - 性能差（锁竞争）                                      │
│      - 容易死锁                                             │
│      - 调试困难                                             │
│                                                              │
│   ❌ 方案2：粗粒度锁                                         │
│      - 性能差（整个世界一把锁）                               │
│      - 吞吐量低                                             │
│                                                              │
│   ✅ 方案3：单线程逻辑                                        │
│      - 正确性完美（没有并发问题）                             │
│      - 需要配合多线程 IO                                      │
│      - 性能通过帧率来保证                                     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 2. 线程模型概述

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        游戏服务器线程模型                          │
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                      主线程                              │  │
│   │                   (游戏逻辑)                             │  │

│   │   ┌──────────┐  ┌──────────┐  ┌──────────┐            │  │
│   │   │  消息     │→ │  ECS     │→ │  场景    │→ ...      │  │
│   │   │  处理     │  │  系统    │  │  更新    │            │  │
│   │   └──────────┘  └──────────┘  └──────────┘            │  │
│   │         ↑                                                   │  │
│   │         │ 无锁                                              │  │
│   │   ┌─────┴───────────────────────────────────────────┐     │  │

│   └─────────────────────────┼─────────────────────────────────┘  │

│   ┌─────────────────────────┼─────────────────────────────────┐  │

│   │                   (网络/数据库)                            │  │
│   │                                                          │  │
│   │   ┌────────────────────────────────────────────────────┐ │  │
│   │   │        IOCP 完成端口                               │ │  │

│   │   ┌─────────────────────┴────────────────────────────┐  │  │
│   │   │ 接收完成 ──▶ 放入接收队列 ──▶ 主线程消费         │  │  │
│   │   └──────────────────────────────────────────────────┘  │  │
│   │                         │                               │  │
│   │   ┌─────────────────────┴────────────────────────────┐  │  │
│   │   │ 发送队列 ───▶ IO 线程发送 ──▶ IOCP 通知完成      │  │  │
│   │   └──────────────────────────────────────────────────┘  │  │
│   │                                                          │  │
│   └──────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

线程数量配置（建议值）：

┌──────────────────────────────────────────────────────────────┐
│                      线程配置                                  │
├─────────────────────┬────────────────────────────────────────┤
│     线程类型         │           数量                          │
├─────────────────────┼────────────────────────────────────────┤
│ 主线程（逻辑）       │  1 个（固定）                          │
├─────────────────────┼────────────────────────────────────────┤
│ IO 线程（IOCP）      │  CPU 核心数 × 2（建议 4-16 个）        │
├─────────────────────┼────────────────────────────────────────┤
│ 数据库线程           │  CPU 核心数 × 2（建议 4-16 个）        │
├─────────────────────┼────────────────────────────────────────┤
│ 日志线程             │  1 个（独立）                           │
├─────────────────────┼────────────────────────────────────────┤
│ 定时器线程           │  1 个（独立）                           │
└─────────────────────┴────────────────────────────────────────┘
```

### 2.2 为什么这样设计？

```
设计原则：

1️⃣ 主线程单线程：游戏逻辑不需要锁
   └─ 所有共享状态由主线程独占
   └─ 避免了 90% 的并发问题

2️⃣ IO 操作全部异步化：
   └─ 网络收发交给 IO 线程
   └─ 数据库操作交给 DB 线程池
   └─ 主线程只负责逻辑计算

3️⃣ 生产者-消费者模式：
   └─ IO 线程是生产者，把消息放入队列
   └─ 主线程是消费者，从队列取出消息
   └─ 队列是线程安全的（Channel<T>）

4️⃣ 帧循环固定更新：
   └─ 每帧固定执行一次所有逻辑
   └─ 帧率可配置（20Hz 省电 ~ 240Hz 极致）
   └─ 保证所有玩家看到的游戏状态一致
```

```

│   │                                                          │  │
│   │  优点：无锁、单帧逻辑完整、延迟可预测                      │  │
```

## 3. 核心组件

### 3.1 Channel（消息队列）

```csharp
/// <summary>
/// Channel - .NET 高性能线程安全队列
/// 
/// 工作原理：
/// 
/// IO 线程（生产者）                    主线程（消费者）
///       │                                   ▲
///       │                                   │
///       ▼                                   │
///  ┌─────────┐                             │
///  │ Channel │ ─────────────────────────────┤
///  └─────────┘                             │
///       │                                   │
///       ▼                                   │
///  Write(data)                             │
///                                          │
///                                          │
/// 
/// 特点：
/// 1. 无锁实现（基于 CAS）
/// 2. 生产者和消费者可以同时操作
/// 3. 支持异步等待（async/await）
/// 4. 内存安全，不会溢出
/// 
/// 对比 BlockingCollection：
/// - BlockingCollection 有锁
/// - Channel 无锁，性能更好
/// - Channel 是 .NET Core 2.0+ 推荐的
/// </summary>
public class MessageChannel<T>
{
    // ============================================================
    // Channel 的内部实现
    // ============================================================
    
    /// <summary>
    /// Channel 是 .NET 提供的泛型异步队列
    /// 
    /// CreateUnbounded：创建无界队列（无限容量）
    /// 
    /// 选项：
    /// - SingleReader = true：只有一个消费者（我们的主线程）
    /// - SingleWriter = false：有多个生产者（多个 IO 线程）
    /// 
    /// 这样设置可以获得最佳性能
    /// </summary>
    private readonly Channel<T> _channel;
    
    // ============================================================
    // 构造函数
    // ============================================================
    
    /// <summary>
    /// 创建消息队列
    /// 
    /// 容量策略：
    /// - capacity = 0：无界队列（无限容量）
    /// - capacity > 0：有界队列，队列满时丢弃最旧的消息
    /// 
    /// 单消费者（主线程）、多生产者（IO 线程）
    /// </summary>
    public MessageChannel(int capacity = 0)
    {
        if (capacity <= 0)
        {
            _channel = Channel.CreateUnbounded<T>(new UnboundedChannelOptions
            {
                SingleReader = true,
                SingleWriter = false,
                AllowSynchronousContinuations = false
            });
        }
        else
        {
            _channel = Channel.CreateBounded<T>(new BoundedChannelOptions(capacity)
            {
                SingleReader = true,
                SingleWriter = false,
                FullMode = BoundedChannelFullMode.DropOldest,
                AllowSynchronousContinuations = false
            });
        }
        _capacity = capacity;
    }
    
    // ============================================================
    // 属性
    // ============================================================
    
    /// <summary>
    /// 当前队列中的消息数
    /// </summary>
    public int Count { get; }
    
    /// <summary>
    /// 队列容量（0 = 无界）
    /// </summary>
    public int Capacity { get; }
    
    /// <summary>
    /// 队列是否为空
    /// </summary>
    public bool IsEmpty { get; }
    
    // ============================================================
    // 公共方法 - 写入
    // ============================================================
    
    /// <summary>
    /// 同步写入消息
    /// 
    /// 线程安全，多个线程可以同时调用
    /// 
    /// 返回值：
    /// - true：写入成功
    /// - false：写入失败（通常是队列满）
    /// </summary>
    public bool Write(T item)
    {
        if (_channel.Writer.TryWrite(item))
        {
            Interlocked.Increment(ref _count);
            return true;
        }
        return false;
    }
    
    /// <summary>
    /// 异步写入消息
    /// 
    /// 适用场景：队列满时需要等待
    /// </summary>
    public ValueTask<bool> WriteAsync(T item, CancellationToken cancellationToken = default)
    {
        if (_channel.Writer.TryWrite(item))
            return new ValueTask<bool>(true);

        return WriteAsyncSlowPath(item, cancellationToken);
    }

    private async ValueTask<bool> WriteAsyncSlowPath(T item, CancellationToken cancellationToken)
    {
        try
        {
            await _channel.Writer.WriteAsync(item, cancellationToken);
            return true;
        }
        catch (ChannelClosedException)
        {
            return false;
        }
    }

    /// <summary>
    /// 批量写入消息
    /// </summary>
    public int WriteRange(IEnumerable<T> items)
    {
        var count = 0;
        foreach (var item in items)
        {
            if (_channel.Writer.TryWrite(item))
            {
                Interlocked.Increment(ref _count);
                count++;
            }
        }
        return count;
    }
    
    // ============================================================
    // 公共方法 - 读取
    // ============================================================
    
    /// <summary>
    /// 尝试读取一条消息
    /// 
    /// 同步读取：
    /// - 如果队列有消息，立即返回
    /// - 如果队列为空，返回 false
    /// 
    /// 这是推荐的方法
    /// 因为主线程每帧都会轮询
    /// </summary>
    public bool TryRead(out T? item)
    {
        if (_channel.Reader.TryRead(out item))
        {
            Interlocked.Decrement(ref _count);
            return true;
        }
        return false;
    }
    
    /// <summary>
    /// 取出队列中所有消息（一次性清空队列）
    /// 
    /// 主线程每帧调用一次，取出所有待处理的消息
    /// </summary>
    public List<T> DrainAll()
    {
        var results = new List<T>();
        while (_channel.Reader.TryRead(out var item))
        {
            Interlocked.Decrement(ref _count);
            results.Add(item);
        }
        return results;
    }
    
    /// <summary>
    /// 取出最多 N 条消息
    /// 
    /// 用于限制单帧处理数量，防止帧率抖动过大
    /// </summary>
    public List<T> DrainUpTo(int maxCount)
    {
        var results = new List<T>(Math.Min(maxCount, 64));
        for (var i = 0; i < maxCount; i++)
        {
            if (_channel.Reader.TryRead(out var item))
            {
                Interlocked.Decrement(ref _count);
                results.Add(item);
            }
            else
                break;
        }
        return results;
    }
    
    // ============================================================
    // 公共方法 - 生命周期
    // ============================================================
    
    /// <summary>
    /// 标记完成（不再接收新消息）
    /// </summary>
    public void Complete()
    {
        _channel.Writer.Complete();
    }
    
    /// <summary>
    /// 清空队列（丢弃所有消息）
    /// </summary>
    public void Clear()
    {
        while (_channel.Reader.TryRead(out _))
        {
            Interlocked.Decrement(ref _count);
        }
    }
}

/// <summary>
/// 消息通道工厂
/// 
/// 提供简化的创建方法
/// </summary>
public static class MessageChannel
{
    /// <summary>
    /// 创建一个无界消息队列（容量无限）
    /// </summary>
    public static MessageChannel<T> CreateUnbounded<T>()
    {
        return new MessageChannel<T>(0);
    }
    
    /// <summary>
    /// 创建一个有界消息队列（最大容量）
    /// </summary>
    public static MessageChannel<T> CreateBounded<T>(int capacity)
    {
        return new MessageChannel<T>(capacity);
    }
}
```

### 3.2 GameScheduler（主循环调度器）

```csharp
/// <summary>
/// 游戏调度器配置
/// </summary>
public class GameSchedulerOptions
{
    /// <summary>
    /// 目标帧率（每秒帧数）
    /// 默认：20 Hz（省电且稳定）
    /// </summary>
    public int TargetFps { get; set; } = 20;

    /// <summary>
    /// 是否启用 FPS 统计
    /// 默认：启用
    /// </summary>
    public bool EnableFpsStats { get; set; } = true;

    /// <summary>
    /// 是否启用帧时间统计
    /// 默认：启用
    /// </summary>
    public bool EnableFrameTimeStats { get; set; } = true;
}

/// <summary>
/// 帧事件参数
/// </summary>
public readonly struct FrameEventArgs
{
    /// <summary>
    /// 帧号（从 1 开始递增）
    /// </summary>
    public long FrameNumber { get; }

    /// <summary>
    /// 本帧 delta time（秒）
    /// 距离上一帧的时间差
    /// </summary>
    public float DeltaTime { get; }

    /// <summary>
    /// 本帧实际执行时间（毫秒）
    /// 用于性能监控
    /// </summary>
    public double FrameTimeMs { get; }
}

/// <summary>
/// 游戏主循环调度器
///
/// 职责：
/// 1. 管理固定帧率的游戏循环
/// 2. 协调所有系统按顺序执行
/// 3. 提供帧开始/结束事件，业务层可以挂接逻辑
/// 4. 统计和监控性能（FPS、帧时间等）
///
/// 帧率说明：
/// - 20Hz  → 每帧 50ms   （省电，适合日常运营）
/// - 60Hz  → 每帧 16.67ms（标准，适合大多数游戏）
/// - 144Hz → 每帧 6.94ms （竞技模式，PVP）
/// - 240Hz → 每帧 4.17ms （极致，ECS 性能要求高）
/// </summary>
public class GameScheduler
{
    // 配置（通过 GameSchedulerOptions 传入）
    private readonly GameSchedulerOptions _options;
    private readonly double _frameIntervalMs;  // 1000 / TargetFps

    // 运行状态
    private volatile bool _isRunning;
    private long _frameCount;

    // 时间控制
    private readonly Stopwatch _stopwatch;
    private long _lastFrameTicks;
    private DateTime _startTime;

    // FPS 统计（最近 100 帧，内部私有）
    private int _fpsCounter;
    private long _fpsLastUpdateTicks;
    private double _currentFps;
    private readonly double[] _frameTimeWindow = new double[100];

    // 事件 —— 事件签名使用 FrameEventArgs
    public event Action<FrameEventArgs>? OnFrameStart;   // 帧开始：读取输入、同步状态
    public event Action<FrameEventArgs>? OnUpdate;        // 主逻辑：ECS 系统、AI、物理
    public event Action<FrameEventArgs>? OnFrameEnd;      // 帧结束：发送消息、持久化、统计

    // 公共属性
    public int TargetFps { get; }      // 目标帧率
    public double CurrentFps { get; }  // 当前实际 FPS
    public long FrameNumber { get; }   // 当前帧号
    public bool IsRunning { get; }     // 是否正在运行
    public DateTime StartTime { get; } // 服务器启动时间
    public double UptimeSeconds { get; } // 运行时间（秒）

    // 构造函数
    public GameScheduler(int targetFps = 20)
    public GameScheduler(GameSchedulerOptions options)

    // 生命周期
    public void Start()       // 启动主循环（阻塞当前线程）
    public void Stop()        // 停止主循环
}

/// <summary>
/// GameScheduler 扩展方法
/// 提供更便捷的注册和启动方式
/// </summary>
public static class GameSchedulerExtensions
{
    // 异步启动（不阻塞调用线程）
    public static Task StartAsync(this GameScheduler scheduler)

    // 便捷注册 Update 事件
    public static void RegisterUpdate(this GameScheduler scheduler, Action<FrameEventArgs> action)
}
```

```
│  Handler 处理消息（主线程）：                                  │
│  await Database.QueryAsync(playerData);  // ❌ 阻塞主线程！  │

│  - 如果数据库查询慢（100ms），主线程要等 100ms               │

│  - 其他玩家的消息无法处理                                      │

│  4. 返回（不等待）                                           │

│  DB 线程池（异步）：                                          │

│  3. 发送响应给客户端                                          │
│                                                              │
│  结果：                                                       │
│  - 主线程永远不阻塞                                           │

│                                                              │
```

## 5. 同步机制

### 5.1 何时需要加锁？

```
单线程逻辑的好处：
└─ 整个主循环只有一个线程
└─ 不需要任何锁

但有些情况下还是需要锁：

1️⃣ Session 字典访问
   └─ IO 线程可能会添加/删除 Session
   └─ 主线程可能会遍历 Session
   └─ 需要 ConcurrentDictionary

2️⃣ 全局配置修改
   └─ GM 后台可能修改配置
   └─ 主线程读取配置
   └─ 需要 volatile 或 ReaderWriterLock

3️⃣ 日志
   └─ 多线程写日志
   └─ 建议用 Serilog，它内部已经有锁
```

### 5.2 线程安全的集合

```csharp
/// <summary>
/// 线程安全的集合选择指南
/// 
/// 场景1：多线程写入，单线程读取
/// └─ MessageChannel<T> ✅（我们的主力）
/// └─ ConcurrentQueue<T> ✅
/// 
/// 场景2：多线程读写，需要按键查找
/// └─ ConcurrentDictionary<K, V> ✅
/// 
/// 场景3：多线程写入，多线程读取，需要按键查找
/// └─ ConcurrentDictionary<K, V> ✅
/// └─ 如果读远多于写，可以用 ReaderWriterLockSlim
/// 
/// 场景4：需要按优先级排序
/// └─ ConcurrentPriorityQueue<T>（.NET 6+）
/// └─ 或者 SortedDictionary + lock
/// </summary>
public class ThreadSafeCollections
{
    private readonly System.Collections.Concurrent.ConcurrentDictionary<long, object> _sessions = new();

    /// <summary>
    /// 当前在线 Session 数量
    /// </summary>
    public int SessionCount => _sessions.Count;

    /// <summary>
    /// 添加或更新一个 Session
    /// </summary>
    public void AddOrUpdateSession(long connectionId, object session)
    {
        _sessions.AddOrUpdate(connectionId, session, (id, old) => session);
    }

    /// <summary>
    /// 移除一个 Session
    /// </summary>
    public bool TryRemoveSession(long connectionId)
    {
        return _sessions.TryRemove(connectionId, out _);
    }

    /// <summary>
    /// 尝试获取一个 Session
    /// </summary>
    public bool TryGetSession(long connectionId, out object? session)
    {
        return _sessions.TryGetValue(connectionId, out session);
    }

    /// <summary>
    /// 获取所有 Session 的快照（返回数组）
    /// </summary>
    public object[] GetAllSessions()
    {
        return _sessions.Values.ToArray();
    }
}
```

```csharp
/// <summary>
/// 线程安全的字典（基于 ConcurrentDictionary）
///
/// 这是游戏服务器最常用的集合类型：
/// - 玩家 Session 管理（ConnectionId -> Session）
/// - 角色信息缓存（PlayerId -> PlayerData）

    /// 如果键已存在，直接返回现有值
    /// </summary>

    }
}
```

```csharp
/// <summary>
/// 原子计数器
///
/// 使用 Interlocked 实现无锁的原子增减
/// 用于：玩家人数、消息计数等统计
/// </summary>
public class AtomicCounter
{
    private long _value;

    /// <summary>
    /// 当前值
    /// </summary>
    public long Value => Interlocked.Read(ref _value);

    /// <summary>
    /// 自增 1
    /// </summary>
    public long Increment()
    {
        return Interlocked.Increment(ref _value);
    }

    /// <summary>
    /// 自减 1
    /// </summary>
    public long Decrement()
    {
        return Interlocked.Decrement(ref _value);
    }

    /// <summary>
    /// 增加指定值
    /// </summary>
    public long Add(long value)
    {
        return Interlocked.Add(ref _value, value);
    }

    /// <summary>
    /// 设置新值（返回旧值）
    /// </summary>
    public long Set(long newValue)
    {
        return Interlocked.Exchange(ref _value, newValue);
    }

    /// <summary>
    /// 重置为 0（返回重置前的值）
    /// </summary>
    public long Reset()
    {
        return Set(0);
    }
}
```

## 7. 使用示例

### 7.1 启动服务器（带线程模型）

```csharp
/// <summary>
/// 服务器启动示例
///
/// 展示如何使用线程模型：
/// 1. 创建各种 MessageChannel
/// 2. 启动 IO 线程池
/// 3. 启动 DB 线程池
/// 4. 启动主循环
/// </summary>
public class ServerStarter
{
    // 业务方自行维护各种 MessageChannel（ThreadSafeCollections 不暴露队列）
    private static readonly MessageChannel<(object Session, IMessage Message)> NetworkChannel = new();
    private static readonly MessageChannel<(object Session, byte[] Data)> SendChannel = new();
    private static readonly MessageChannel<DbRequest> DbRequestChannel = new();
    private static readonly MessageChannel<DbResponse> DbResponseChannel = new();

    public static async Task Main(string[] args)
    {
        // ----------------------------------------
        // 1. 初始化 Session 管理
        // ----------------------------------------
        var threadSafe = new ThreadSafeCollections();

        // ----------------------------------------
        // 2. 创建 IOCP 服务器
        // ----------------------------------------
        var server = new TcpServer(new TcpServerOptions
        {
            Port = 9000,
            Backlog = 200
        });

        // 配置消息回调 - 放入主线程队列
        //    Process incoming messages through your message channel.
        server.OnMessageReceived += (session, message) =>
        {
            // ✅ 这里只是放入队列，不阻塞
            NetworkChannel.Write((session, message));
        };

        // ----------------------------------------
        // 3. 启动主循环
        // ----------------------------------------
        var scheduler = new GameScheduler(new GameSchedulerOptions
        {
            TargetFps = 240  // 极致模式（4.17ms/帧）
        });

        // 配置每帧的处理（事件参数使用 FrameEventArgs，通过 args 访问 FrameNumber/DeltaTime/FrameTimeMs）
        scheduler.OnFrameStart += args =>
        {
            // 处理网络消息
            foreach (var (session, message) in NetworkChannel.DrainAll())
            {
                MessageRouter.Route(session, message);
            }

            // 可选：使用 args 追踪帧信息
            // var frame = args.FrameNumber;
            // var dt = args.DeltaTime;
            // var ftMs = args.FrameTimeMs;
        };

        scheduler.OnFrameEnd += args =>
        {
            // 处理发送队列
            foreach (var (session, data) in SendChannel.DrainAll())
            {
                session.SendAsync(data);
            }

            // 处理 DB 响应
            foreach (var response in DbResponseChannel.DrainAll())
            {
                // 执行回调（在主线程执行）
                response.Continuation?.Invoke(response);
            }
        };

        // ----------------------------------------
        // 4. 启动 DB 线程池
        // ----------------------------------------
        var dbThreadCount = Math.Max(4, Environment.ProcessorCount * 2);
        var dbThreads = new List<Thread>();

        for (int i = 0; i < dbThreadCount; i++)
        {
            var thread = new Thread(() => DbWorker());
            thread.Name = $"DB-Worker-{i}";
            thread.IsBackground = true;
            thread.Start();
            dbThreads.Add(thread);
        }

        // ----------------------------------------
        // 5. 启动服务器
        // ----------------------------------------
        server.Start();
        scheduler.Start();
    }

    /// <summary>
    /// 数据库工作线程
    ///
    /// 从 DB 请求队列取任务，执行，然后放入响应队列
    /// </summary>
    private static void DbWorker()
    {
        Logger.Info($"DB Worker 启动: {Thread.CurrentThread.Name}");

        while (true)
        {
            if (DbRequestChannel.TryRead(out var request))
            {
                try
                {
                    // 执行数据库操作
                    var result = ExecuteDbRequest(request);

                    // 放入响应队列
                    DbResponseChannel.Write(result);
                }
                catch (Exception ex)
                {
                    Logger.Error($"DB 操作失败: {ex}");

                    DbResponseChannel.Write(new DbResponse
                    {
                        Success = false,
                        Error = ex.Message
                    });
                }
            }
        }
    }

    /// <summary>
    /// 执行数据库请求
    ///
    /// 这是一个模拟方法，实际会调用真实的数据库
    /// </summary>
    private static DbResponse ExecuteDbRequest(DbRequest request)
    {
        // 模拟数据库操作耗时
        Thread.Sleep(10);

        return new DbResponse
        {
            Success = true,
            Data = "result"
        };
    }
}
```

### 7.2 编写线程安全的 Handler

```csharp
/// <summary>
/// 消息处理器基类
///
/// 线程安全说明：
/// 1. Handle 方法在主线程执行
/// 2. 可以安全访问 ECS World
/// 3. 可以安全访问共享数据
/// 4. 不要在这里做同步 IO
/// </summary>
public abstract class MessageHandler<TMessage> : IMessageHandler<TMessage>
    where TMessage : IMessage
{
    /// <summary>
    /// 异步数据库请求队列
    /// 注意：ThreadSafeCollections 不再公开队列属性，
    /// 业务方需通过 MessageChannel<T> 自行维护队列。
    /// </summary>
    protected static MessageChannel<DbRequest> DbQueue { get; } = new MessageChannel<DbRequest>();

    /// <summary>
    /// 发送队列
    /// </summary>
    protected static MessageChannel<(object Session, byte[] Data)> SendQueue { get; }
        = new MessageChannel<(object Session, byte[] Data)>();

    /// <summary>
    /// 处理消息
    ///
    /// 注意：这个方法在主线程执行
    /// </summary>
    public abstract Task Handle(object session, TMessage message);

    /// <summary>
    /// 辅助方法：发送响应给客户端
    ///
    /// 消息会被放入发送队列，由 IO 线程发送。
    /// 使用 MessageChannel.WriteAsync 代替 Channel.Writer.WriteAsync。
    /// </summary>
    protected async ValueTask SendAsync(object session, IMessage message)
    {
        var data = MessageSerializer.Serialize(message);
        await SendQueue.WriteAsync((session, data));
    }

    /// <summary>
    /// 辅助方法：发起异步数据库请求
    ///
    /// 请求会被放入 DB 队列，由 DB 线程执行
    /// 回调会在主线程执行
    /// </summary>
    protected void QueryDatabase<TRequest, TResponse>(
        TRequest request,
        Action<TResponse> onComplete)
    {
        DbQueue.Write(new DbRequest
        {
            Type = typeof(TRequest).Name,
            Data = request,
            Continuation = (response) => onComplete((TResponse)response.Data)
        });
    }
}

/// <summary>
/// 移动消息处理器示例
/// </summary>
public class MoveHandler : MessageHandler<C2S_Move>
{
    private readonly World _world;
    
    public MoveHandler(World world)
    {
        _world = world;
    }
    
    public override Task Handle(Session session, C2S_Move message)
    {
        // ----------------------------------------
        // ✅ 在主线程执行，安全无锁
        // ----------------------------------------
        
        // 获取玩家的 ECS Entity
        if (!_world.TryGetEntity(session.PlayerId, out var entity))
        {
            return Task.CompletedTask;
        }
        
        // 获取位置组件
        ref var position = ref _world.GetComponent<PositionComponent>(entity);
        
        // 计算新位置
        position.X = message.TargetX;
        position.Y = message.TargetY;
        position.Z = message.TargetZ;
        
        // ✅ 可以安全访问其他实体的数据
        // 因为没有其他线程在修改
        var nearbyEntities = _world.QueryEntities<PositionComponent>(entity);
        foreach (var nearby in nearbyEntities)
        {
            // 广播移动给周围玩家
            var nearbyPos = _world.GetComponent<PositionComponent>(nearby);
            var nearbySession = GetSession(nearby);
            nearbySession?.SendAsync(new S2C_EntityMove
            {
                EntityId = session.PlayerId,
                X = position.X,
                Y = position.Y,
                Z = position.Z
            });
        }
        
        return Task.CompletedTask;
    }
}
```

## 6. 雪花算法时间回拨处理 (v2 优化)

### 6.1 问题背景

雪花算法依赖时间戳生成唯一 ID。当系统时间被调整（如 NTP 同步、时钟漂移）导致时间回拨时：

```
时间 1000: 生成 ID = (1000 << 22) | (machineId << 12) | sequence
时间 990:  生成 ID = (990  << 22) | (machineId << 12) | sequence  ← ID 重复!
```

### 6.2 解决方案

**分级处理策略**：

| 回拨程度 | 策略 | 日志级别 |
|---------|------|---------|
| ≤ 5ms | 自旋等待（busy-wait），等待时间追上 | Debug |
| > 5ms | 记录错误日志，强制使用当前时间戳（⚠️ 可能产生 ID 重复） | Error |

### 6.3 实现要点

```csharp
// 实际源码的时间回拨处理（2 档，不是文档描述的 3 档）
private long HandleClockRollback(long lastTimestamp, long currentTimestamp)
{
    long diff = lastTimestamp - currentTimestamp;

    if (diff <= 5)
    {
        // 小回拨（≤5ms）：自旋等待
        Thread.SpinWait((int)diff * 1000);
        return lastTimestamp;
    }
    else
    {
        // 大回拨（>5ms）：记录 Error，强制使用当前时间
        Logger.Error("Snowflake", "严重时间回拨 {0}ms，强制使用当前时间戳", diff);
        return currentTimestamp;  // ⚠️ 可能产生 ID 重复
    }
}
```

> ⚠️ **注意**：文档旧版本描述了 3 档策略（≤10ms / ≤1000ms / >1000ms），但实际源码只有 **2 档**（≤5ms / >5ms）。超过 5ms 的回拨直接报错并强制前进，不会使用扩展序列号。

### 6.4 监控建议

```csharp
// 记录时间回拨次数
MetricsCollector.Instance.RegisterCounter("snowflake.clock_rollback");
MetricsCollector.Instance.IncrementCounter("snowflake.clock_rollback", 1);
```
