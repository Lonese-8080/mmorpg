## 1. 快速开始

### 1.1 启动方式

```bash
# 方式 A：直接运行（开发）
dotnet run --project samples/MMORPG.Sample -c Release

# 方式 B：本地 Docker Compose
docker compose up -d

# 方式 C：预构建镜像
docker run -d --name mmorpg \
  -p 9000:9000 -p 8080:8080 -p 9091:9091 \
  -e MMORPG_CONFIG_KEY="$(openssl rand -hex 32)" \
  -v $(pwd)/logs:/app/logs \
  ghcr.io/<owner>/mmorpg-framework:2.2.0
```

### 1.2 必须的环境变量

| 变量 | 是否必填 | 说明 |
|------|----------|------|
| `MMORPG_CONFIG_KEY` | **是（生产）** | 配置文件敏感字段解密的主密钥 |
| `Logging__MinLevel` | 否 | 覆盖配置文件中的日志级别 |
| `Database__ConnectionString` | 否 | 覆盖配置文件中的连接串 |
| `Network__Port` | 否 | 覆盖监听端口 |

### 1.3 端口规划

| 端口 | 用途 | 必须暴露？ |
|------|------|------------|
| 9000 | TCP 游戏端口 | ✅ 玩家侧 |
| 8080 | /health /ready /alive | ✅ 集群 / 负载均衡 |
| 9091 | Prometheus 指标 | ✅ Prometheus 抓取 |

## 2. 配置管理

### 2.1 配置文件结构

```jsonc
// appsettings.production.json
{
  "SchemaVersion": "2.2.0",
  "Server":      { "ServerId": 1, "ServerName": "S1", "Type": "Game" },
  "Performance": { "TargetFps": 60 },
  "Network":     { "Port": 9000, "MaxConnections": 10000 },
  "Database":    { "Type": "MySql", "ConnectionString": "ENC[v1]:..." }
}
```

### 2.2 敏感字段加密

加密工具（任意 C# REPL）：

```csharp
ConfigEncryptor.SetMasterKey(Environment.GetEnvironmentVariable("MMORPG_CONFIG_KEY"));
Console.WriteLine(ConfigEncryptor.Encrypt("my-db-password"));
```

替换后 `ConfigurationLoader.Load` 会自动识别 `ENC[v1]:` 前缀并解密。

**注意**：密钥泄漏即等于明文密码泄漏，请使用 Vault / KMS。

### 2.3 配置热更新

修改 JSON 文件后，框架会在默认 1 秒内检测到变更并触发 `ConfigChanged` 事件。
**不支持热更新的字段**（需重启）：

- `Network.Port`（监听套接字）
- `Database.Type`（连接池语义）
- `Performance.TargetFps`（调度器节拍）

### 2.4 通过环境变量覆盖

```bash
# 覆盖单字段
export Server_ServerId=42
export Logging_MinLevel=Warning

# 容器场景（使用双下划线）
docker run -e Server__ServerId=42 -e Logging__MinLevel=Warning ...
```

## 3. 健康检查与可观测性

### 3.1 HTTP 端点

| 端点 | 返回码 | 用途 |
|------|--------|------|
| `/alive` | 200 | 进程存活探针（轻量、不依赖子系统） |
| `/ready` | 200/503 | 就绪探针（依赖 DB / 缓存等关键依赖） |
| `/health` | 200/503 | 详细健康报告（JSON） |

调用示例：

```bash
curl -i http://server:8080/alive    # 200
curl -i http://server:8080/ready    # 200 / 503
curl  http://server:8080/health | jq
```

返回 JSON 形如：

```json
{
  "status": "Healthy",
  "totalDurationMs": 12.3,
  "entries": {
    "tcp_listener": { "status": "Healthy", "durationMs": 0.1 },
    "memory":       { "status": "Healthy", "durationMs": 0.5 }
  }
}
```

### 3.2 Prometheus 抓取

`/metrics` 暴露 Prometheus 文本格式。可在 `prometheus.yml` 中配置：

```yaml
scrape_configs:
  - job_name: 'mmorpg'
    scrape_interval: 15s
    static_configs:
      - targets: ['mmorpg:9091']
```

关键指标：

| 指标 | 类型 | 含义 |
|------|------|------|
| `mmorpg_active_sessions` | Gauge | 当前活跃会话数 |
| `mmorpg_messages_received_total` | Counter | 累计接收消息数 |
| `mmorpg_messages_sent_total` | Counter | 累计发送消息数 |
| `mmorpg_message_retry_total` | Counter | 累计重试次数 |
| `mmorpg_dead_letter_total` | Counter | 死信累计 |
| `mmorpg_rpc_duration_seconds` | Histogram | RPC 处理时延 |
| `mmorpg_memory_working_set_mb` | Gauge | 工作集内存 MB |

### 3.3 Grafana 仪表盘建议

最小可用面板：

1. **P99 消息处理时延**（`histogram_quantile(0.99, mmorpg_rpc_duration_seconds_bucket)`）
2. **活跃会话数趋势**
3. **重试率**（`rate(mmorpg_message_retry_total[5m])`）
4. **死信率**（同上但针对 dead_letter）
5. **工作集内存 + GC 暂停**（dotnet 内部指标）

## 4. 故障排查 Runbook

### 4.1 服务无法启动

| 症状 | 排查步骤 |
|------|----------|
| 退出码 1 / 立即崩溃 | 1. 检查 `MMORPG_CONFIG_KEY` 是否设置（若 JSON 中存在 `ENC[v1]:`） 2. 验证 SchemaVersion 是否为 SemVer 3 段格式 3. 端口是否被占用：`netstat -ano \| findstr 9000` |
| 配置校验失败 | 查看启动日志中的 `ValidationResult` 行，逐项修复 JSON |
| 缺少依赖 | `dotnet list package --vulnerable` 检查；CI 阶段会扫描 CVE |

### 4.2 玩家连接失败

```bash
# 1. 端口是否对外暴露？
nc -zv <server-ip> 9000

# 2. 服务端是否在监听？
ss -ltnp | grep 9000

# 3. /alive 与 /ready 是否健康？
curl http://<server>:8080/alive
curl http://<server>:8080/ready
```

若 `/ready` 返回 503：检查依赖项（如数据库连接）状态。
若 `/alive` 返回 200 但 `/ready` 503：服务在降级运行，可接受流量但部分功能不可用。

### 4.3 消息处理时延飙升

按下列顺序排查：

1. **P99 vs P50**：若仅 P99 飙升，多为 GC / 锁竞争；若 P50 都升高，多为下游慢。
2. **查看死信**：`mmorpg_dead_letter_total` 若上涨，检查重试逻辑与下游可用性。
3. **查看活跃会话**：`mmorpg_active_sessions` 是否触达 `Network.MaxConnections` 上限。
4. **GC 暂停**：`dotnet-gcdump` + `dotnet-trace` 采样 30 秒后分析 `gc_pause` 段。

### 4.4 内存持续增长

| 现象 | 可能原因 | 处置 |
|------|----------|------|
| 缓增不释放 | 分配器碎片 / 池化对象未归还 | `dotnet-gcdump` 抓快照后比 HeapDiff |
| 突增不释放 | 事件订阅未取消 / 静态集合累积 | 检查 `+=` 是否有对应 `-=` |
| OOM Killed | 容器限制过低 | 调高 `memory.limit` + 启用 GC 调优 |

### 4.5 容器被 OOM Kill

```bash
dmesg -T | grep -i "killed process"
docker inspect mmorpg --format '{{.State.OOMKilled}}'
```

调整方案：

```yaml
# docker-compose.yml
deploy:
  resources:
    limits:
      memory: 6144M
    reservations:
      memory: 2048M
```

同时调整 GC：

```bash
DOTNET_gcServer=1              # 服务器 GC
DOTNET_GCConserveMemory=5      # 0=平衡 5=保守 9=最保守
```

## 5. 性能调优手册

### 5.1 调优原则

> **永远先测量，再优化**。所有"调优"建议均基于典型场景，需结合实际 profile 验证。

### 5.2 操作系统层（Linux）

```bash
# /etc/sysctl.d/99-mmorpg.conf
# 文件描述符
fs.file-max = 2097152
fs.nr_open  = 1048576

# TCP 缓冲区
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216
net.ipv4.tcp_rmem = 4096 87380 16777216
net.ipv4.tcp_wmem = 4096 65536 16777216

# TIME_WAIT 复用
net.ipv4.tcp_tw_reuse = 1
net.ipv4.tcp_max_tw_buckets = 262144

# 端口范围
net.ipv4.ip_local_port_range = 1024 65535

# 连接队列
net.core.somaxconn = 65535
net.ipv4.tcp_max_syn_backlog = 65535

# 透明大页（数据库推荐关，游戏推荐开）
# echo madvise > /sys/kernel/mm/transparent_hugepage/enabled
```

应用：

```bash
sudo sysctl --system
ulimit -n 1048576
```

### 5.3 .NET 运行时调优

```bash
# GC（默认 Server 模式）
DOTNET_gcServer=1
DOTNET_GCConserveMemory=5            # 0~9，越高越省内存越低吞吐
DOTNET_gcHeapHardLimit=0             # 不限制；或设为容器内存 75%
DOTNET_gcHeapHardLimitPercent=75

# 线程池
DOTNET_ThreadPool_MinThreads=200     # 避免初始突发时线程饥饿
DOTNET_ThreadPool_MaxThreads=2000

# JIT（预热生产环境可考虑 R2R / ReadyToRun）
DOTNET_TieredCompilation=1
DOTNET_TieredPGO=1
```

### 5.4 框架级调优

#### TcpServer

| 参数 | 建议值 | 说明 |
|------|--------|------|
| `Network.MaxConnections` | 10000~50000 | 取决于单节点内存（每会话 ~16KB） |
| `Network.ReceiveBufferSize` | 8192~65536 | 增大可降低 syscall 次数但增加内存 |
| `Network.SendBufferSize` | 8192~65536 | 同上 |
| `Network.HeartbeatTimeout` | 20s | 心跳间隔的 3~4 倍 |
| `Network.SessionMessageRateLimit` | 100~500 | 0 表示不限；防作弊必须设置 |
| `Network.GlobalMessageRateLimit` | 10000~50000 | 0 表示不限 |
| `Network.MessageRetryCount` | 3 | 配合指数退避 |
| `Network.MessageRetryIntervalMs` | 100 | 退避基数 |

#### SessionPool

| 参数 | 建议值 | 说明 |
|------|--------|------|
| `EnablePool` | true | 生产必须启用 |
| `MinPoolSize` | 200~500 | 预热数量，约等于稳态在线的 10% |
| `MaxPoolSize` | 2000~10000 | 应略大于峰值 |
| `MaxIdleCapacity` | MaxPoolSize/2 | 空闲回收上限 |

#### Logging

| 参数 | 建议值 | 说明 |
|------|--------|------|
| `MinLevel` | Info | 生产；Debug 仅诊断用 |
| `SamplingRate` | 0.1~0.3 | Debug/Warning 受影响；Error/Fatal 始终记录 |
| `SuppressDuplicateThresholdMs` | 1000 | 防刷屏 |
| `EnableNetwork` | true | 接入日志聚合（Vector / Fluent Bit） |

#### Observability

| 参数 | 建议值 | 说明 |
|------|--------|------|
| `MemoryWarningThresholdMB` | 容器限制 × 0.6 | 提前告警 |
| `MemoryCriticalThresholdMB` | 容器限制 × 0.8 | 触发 OOM 前告警 |
| `MemoryCheckIntervalSeconds` | 30 | 检查频率 |

### 5.5 数据库调优

- **连接池**：`MaxPoolSize` 应等于 `MaxConnections / 4` 左右（取决于 DB 端能承载的并发）。
- **超时**：`ConnectionTimeout=10`，`CommandTimeout=30`；慢查询必须通过 SQL 端优化。
- **预编译**：所有热路径 SQL 必须预编译，避免每次发送完整文本。

### 5.6 性能基准对照

通过 `benchmarks/MMORPG.Framework.Benchmarks` 跟踪回归：

```bash
dotnet run --project benchmarks/MMORPG.Framework.Benchmarks -c Release -- --job short
```

基线参考（v2.2.0，Intel Xeon E5-2680v4 / Linux 5.15）：

| 基准 | Median | 备注 |
|------|--------|------|
| Serialize 1KB 消息 | < 800 ns | BinaryWriter |
| Deserialize 1KB 消息 | < 1.2 µs | BinaryReader |
| HeaderParse 4B | < 30 ns | 单一读 |
| Config.Validate() | < 50 µs | ServerConfig 全字段 |
| BoundedChannel<>.WriteAsync | < 200 ns | Bounded=10000 |

> 若性能退化超过 20%，应先回归 Profile 再合并。

## 6. 容量规划

### 6.1 内存估算公式

```
单节点内存 ≈ 基础运行时(200MB)
           + 活跃会话 × 16KB
           + MessageChannel 容量 × 256B
           + GC Heap HardLimit(可配)

例如：10000 在线 → 200 + 160 + 2.5 = ~365MB
```

### 6.2 CPU 估算

经验值：单核 8 字节 IO 完成 + 反序列化 ≈ 150K QPS
10K 在线、单玩家 5 msg/s = 50K msg/s → 约 1/3 核（理想环境）

实际需保留 50% 余量应对峰值与 GC。

### 6.3 网络带宽

```
带宽 = 平均消息大小 × QPS × 8 / 0.7
(0.7 为可用带宽系数)
```

10K 在线 × 5 msg/s × 256B × 8 / 0.7 ≈ 145 Mbps

### 6.4 水平扩展

无状态网关 + 游戏服：
- 网关承担接入 / 加密 / 限流
- 游戏服按地图/线路拆分（ECS Sharding）

## 7. 安全运维

### 7.1 必做项

- [ ] 生产环境**必须**设置 `MMORPG_CONFIG_KEY`
- [ ] 私钥 / 证书走 Vault 注入，**禁止**提交到 Git
- [ ] 启用 TLS（`Network.EnableTls=true` + `CertificateHotReload=true`）
- [ ] 设置 `SessionMessageRateLimit` 与 `GlobalMessageRateLimit`
- [ ] 容器以非 root 运行（已默认 `mmorpg:10001`）
- [ ] 启用只读根文件系统 + tmpfs
- [ ] 限制 Linux capabilities（`--cap-drop=ALL --cap-add=NET_BIND_SERVICE`）

### 7.2 密钥轮换

```bash
# 1. 旧密钥仍持有者：双写新字段名
old_config.json
new_config.json (使用新密钥加密)

# 2. 滚动重启集群，分批切换 MMORPG_CONFIG_KEY 环境变量
# 3. 旧配置文件归档后删除
```

### 7.3 漏洞响应

1. 接收 Dependabot / 内部漏洞通告
2. 在测试环境复现 → 评估影响
3. 升级依赖 → 走 release 分支 → 灰度

## 8. 灾备与回滚

### 8.1 灰度发布

```bash
# 1. 新镜像 tag
docker tag mmorpg:2.2.1 ghcr.io/xxx/mmorpg:2.2.1

# 2. 部署 1 个灰度实例（流量 5%）
docker compose up -d --scale mmorpg-server=2   # 由 LB 按权重切流

# 3. 监控 30 分钟，确认无异常
# 4. 全量滚动：先启新 → 健康检查通过 → 杀旧
```

### 8.2 回滚

```bash
# 1. 切回上一版本镜像
docker compose pull mmorpg-server  # 上一 stable
docker compose up -d

# 2. 配置回滚：保留旧版 JSON 至少 7 天
git tag v2.2.0 config/   # 配合 git 跟踪
```

### 8.3 数据一致性

- 配置热更新有 1s 窗口期，**禁止**在窗口期内触发停机。
- 重启前主动停 `/ready` 探针（让 LB 摘流），再 `SIGTERM`。

|------|------------|

### 9.3 应急命令

```bash
# 1. 抓 GC dump
docker exec mmorpg bash -c 'curl http://localhost:9000/diag/gcdump -o /tmp/gc.gcdump' 2>/dev/null \
  || dotnet-gcdump collect -p 1

# 2. 抓 trace
dotnet-trace collect -p $(pgrep -f MMORPG.Sample) --duration 00:00:30

# 3. 看线程栈
dotnet-dump ps && dotnet-dump collect -p <pid>

# 4. 紧急切只读
docker exec mmorpg touch /app/READONLY_FLAG   # 业务侧识别
```
