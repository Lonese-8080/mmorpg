## 1. 设计原则

### 1.1 核心设计原则

| 原则 | 说明 | 在本框架中的体现 |
|------|------|----------------|
| **高性能优先** | 服务器必须能处理高并发 | IOCP + 对象池 + 零拷贝 |
| **无锁设计** | 避免锁竞争导致的性能瓶颈 | 单线程逻辑 + Channel 队列 |
| **缓存友好** | 充分利用 CPU 缓存 | Archetype 连续内存布局（规划中） |
| **模块化** | 各模块松耦合，可独立测试 | Framework/Core/Game 分层 |
| **可扩展** | 易于添加新功能 | ECS 组件化设计（规划中） |

### 1.2 为什么选择这些原则？

**问题**：MMORPG 服务器最怕什么？

```
答案：卡顿、掉线、崩溃

原因：
1. 玩家数量多（1000-10000+）
2. 每秒消息量大（10000-100000+）
3. 逻辑计算复杂（战斗、AI、广播）
4. 数据一致性要求高（不能丢数据）
```

**我们的解法**：

| 问题 | 解法 |
|------|------|
| 卡顿 | 固定帧预算 + 对象池 + 零分配 |
| 掉线 | 心跳检测 + 自动重连 |
| 崩溃 | 异常隔离 + 日志追踪 |

## 4. 核心数据流

### 4.1 消息处理流程

```
玩家客户端                                          游戏服务器
    │                                                   │
    │  ① 发送消息 (TCP)                                │
    │  ─────────────────────────────────────────────▶ │
    │  消息格式: [长度4B][类型2B][内容N B]              │
    │                                                   │
    │                     ┌────────────────────────────┐│
    │                     │     IOCP 接收完成          ││
    │                     │  ② 放入接收队列            ││
    │                     │     (Channel)             ││
    │                     └────────────┬───────────────┘│
    │                                  │                 │
    │                     ┌────────────▼───────────────┐│
    │                     │     主线程轮询             ││
    │                     │  ③ 从队列取出消息          ││
    │                     └────────────┬───────────────┘│
    │                                  │                 │
    │                     ┌────────────▼───────────────┐│
    │                     │   MessageRouter 消息路由   ││
    │                     │  ④ 根据消息类型分发        ││
    │                     │     到对应 Handler         ││
    │                     └────────────┬───────────────┘│
    │                                  │                 │
    │                     ┌────────────▼───────────────┐│
    │                     │   Handler 处理业务逻辑     ││
    │                     │  ⑤ 修改 ECS World 中的数据 ││
    │                     └────────────┬───────────────┘│
    │                                  │                 │
    │                     ┌────────────▼───────────────┐│
    │                     │   生成响应消息             ││
    │                     │  ⑥ 放入发送队列            ││
    │                     └────────────┬───────────────┘│
    │                                  │                 │
    │  ⑦ 发送响应消息 (TCP)              │                 │
    │  ◀─────────────────────────────────────────────── │
    │                                                   │
```

**关键点说明**：

| 步骤 | 说明 | 为什么这么做 |
|------|------|-------------|
| ① 消息格式 | 长度+类型+内容 | 粘包处理，明确每条消息边界 |
| ② 接收队列 | Channel<T> | 线程安全、无锁、生产者-消费者模式 |
| ③ 主线程轮询 | 每帧检查队列 | 保证逻辑顺序一致，无锁 |
| ④ 消息路由 | HashMap<MsgId, Handler> | O(1) 查找，快速分发 |
| ⑤ 业务逻辑处理 | Handler 处理业务逻辑 | ⏳ 规划中 |
| ⑥ 发送队列 | 另一个 Channel | 解耦逻辑和IO |
| ⑦ 异步发送 | IOCP 通知完成 | 不阻塞主线程 |

### 4.2 游戏帧循环

> 📝 **说明**：ECS 系统执行、场景/AOI 更新、广播消息发送等步骤目前 **⏳ 规划中**，只有框架层（网络消息处理 + 帧循环调度器）已实现。

```
┌─────────────────────────────────────────────────────────────┐
│                      主线程帧循环 (FixedUpdate)               │
│                                                             │
│  每帧执行时间: 50ms (20Hz) 或 16.67ms (60Hz)               │
│                                                             │
│  ┌─────────────────────────────────────────────────────────┐│
│  │  帧开始                                                  ││
│  │    ↓                                                    ││
│  │  处理网络消息队列  ──▶  从 Channel 取出所有消息         ││
│  │    ↓                                                    ││
│  │  消息路由分发    ──▶  调用对应 Handler                  ││
│  │    ↓                                                    ││
│  │  执行 ECS 系统  ──▶  按顺序执行所有 System.Update()     ││
│  │    ↓                                                    ││
│  │  场景管理更新  ──▶  AOI 视野同步                        ││
│  │    ↓                                                    ││
│  │  广播消息发送  ──▶  把要发的消息放入发送队列             ││
│  │    ↓                                                    ││
│  │  帧结束          ──▶  Thread.Sleep 或 Yield              ││
│  └─────────────────────────────────────────────────────────┘│
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**帧预算分配（建议值）**：

```
┌─────────────────────┬────────────┬──────────────────────────┐
│       阶段           │  预算时间   │           说明            │
├─────────────────────┼────────────┼──────────────────────────┤
│ 网络消息处理         │   5ms      │ 从队列取出 + 路由分发      │
├─────────────────────┼────────────┼──────────────────────────┤
│ ECS 系统执行         │   10ms     │ 遍历 + 计算                │
├─────────────────────┼────────────┼──────────────────────────┤
│ 场景/AOI 更新        │   5ms      │ 视野同步                   │
├─────────────────────┼────────────┼──────────────────────────┤
│ 消息广播             │   5ms      │ 打包 + 放入发送队列        │
├─────────────────────┼────────────┼──────────────────────────┤
│ 其他开销             │   5ms      │ 日志、定时器检查等         │
├─────────────────────┼────────────┼──────────────────────────┤
│ 总计                 │   30ms     │ 预留余量，理论帧率 33Hz    │
└─────────────────────┴────────────┴──────────────────────────┘
```

## 5. 技术选型理由

### 5.1 为什么选择 .NET 10？

| 特性 | 说明 | 对游戏的帮助 |
|------|------|-------------|
| **GC 改进** | .NET 10 的 GC 停顿大幅减少，接近 C++ 级别 | 消除卡顿 |
| **source generator** | 编译时生成代码 | 减少运行时反射开销 |
| **AOT 编译** | 提前编译成机器码 | 启动更快，内存更少 |
| **跨平台** | Windows/Linux/macOS | 部署灵活 |
| **生态完善** | 大量成熟库 | 可按需引入第三方库 |

### 5.2 为什么选择 IOCP？

```
IOCP (Input/Output Completion Port) = Windows 高性能 IO 模型

对比其他方案：

┌─────────────┬──────────────┬──────────────┬──────────────────┐
│    方案      │    性能       │   易用性      │    适用场景       │
├─────────────┼──────────────┼──────────────┼──────────────────┤
│ IOCP        │  ★★★★★        │  ★★★☆☆       │ Windows 高性能服务器│
├─────────────┼──────────────┼──────────────┼──────────────────┤
│ epoll       │  ★★★★★        │  ★★★☆☆       │ Linux 高性能服务器│
├─────────────┼──────────────┼──────────────┼──────────────────┤
│ Select      │  ★★☆☆☆        │  ★★★★★       │ 低并发场景        │
├─────────────┼──────────────┼──────────────┼──────────────────┤
│ async/await │  ★★★★☆        │  ★★★★★       │ 中等并发 Web 服务 │
└─────────────┴──────────────┴──────────────┴──────────────────┘

结论：对于 10000+ 连接的游戏服务器，IOCP 是 Windows 平台最优选择
```

### 5.3 序列化 — Google Protobuf

> 📝 **说明**：当前框架使用 **Google Protobuf** 作为消息序列化方案，依赖 `Google.Protobuf` v3.30.2 与 `Grpc.Tools` v2.72.0。`.proto` 文件位于 `protos/` 目录，由 MSBuild 在编译时自动生成 C# 代码到 `Framework.Network.Protobuf` 命名空间。`MessageSerializer` 位于 `Framework.Network` 命名空间，基于 `Google.Protobuf.IMessage` 接口实现 `MergeFrom`/`WriteTo` 零分配 API。

**Protobuf 序列化的当前选择：**

| 特性 | 说明 |
|------|------|
| 位置 | `Framework.Network.MessageSerializer` |
| 协议 | `[长度4B][消息ID 4B][Protobuf消息体 N字节]` |
| 跨语言 | 多语言支持（C#/Java/Go/Python/...） |
| 优势 | 行业标准、协议演进、零分配 API、丰富调试工具 |

### 5.4 为什么选择 ECS (Archetype)？

> 📝 **说明**：ECS (Archetype) 架构目前 **⏳ 规划中**，尚未实现。以下为设计理念介绍。

```
传统 OOP 方式：
┌─────────────────────────────────────┐
│     Player : MonoBehaviour          │
│  ┌─────────────────────────────────┐│
│  │ position: Vector3                ││
│  │ health: int                      ││
│  │ attack: int                     ││
│  │ items: List<Item>               ││
│  │ skills: Skill[]                  ││
│  │ buffManager: BuffManager        ││
│  │ ...（可能有50+字段）             ││
│  │                                 ││
│  │ Update() { /* 移动逻辑 */ }     ││
│  │ TakeDamage() { /* 受伤逻辑 */ } ││
│  │ UseSkill() { /* 技能逻辑 */ }   ││
│  └─────────────────────────────────┘│
└─────────────────────────────────────┘

问题：
1. 不同玩家有不同字段，内存不连续
2. 想遍历"所有有血量的对象"很慢
3. 多态导致虚函数调用开销
4. 难以并行处理

ECS 方式：
┌─────────────────────────────────────┐
│  World                              │
│                                     │
│  Archetype { Position, Health }     │
│  ┌──────────┬──────────┐            │
│  │ Position │  Health  │            │
│  ├──────────┼──────────┤            │
│  │ P1        │ H1       │  ← 实体#1  │
│  ├──────────┼──────────┤            │
│  │ P2        │ H2       │  ← 实体#2  │
│  └──────────┴──────────┘            │
│                                     │
│  Archetype { Position, Health, Atk }│
│  ┌──────────┬──────────┬──────────┐ │
│  │ Position │  Health  │  Attack  │ │
│  ├──────────┼──────────┼──────────┤ │
│  │ P3        │ H3       │ A3       │ │
│  └──────────┴──────────┴──────────┘ │
│                                     │
│  System: MovementSystem              │
│  foreach (有 Position 的 Archetype) ││
│      批量处理位置更新...            │
│                                     │
│  System: BattleSystem               │
│  foreach (有 Health+Attack的Archetype)│
│      批量处理战斗逻辑...            │
└─────────────────────────────────────┘

优势：
1. 相同组件连续内存，CPU 缓存命中率高
2. 批量处理，减少分支预测失败
3. 逻辑和数据分离，更易并行
4. 组件可自由组合，灵活度高
```

## 8. 商业化能力补充

在 v2 商业化打磨阶段，框架新增以下企业级能力：

### 8.1 指标监控 (Metrics)

- **三类指标**：Counter（计数器）、Gauge（仪表盘）、Histogram（直方图）
- **单例注册中心**：MetricsCollector 全局单例，支持线程安全的指标注册和查询
- **关键业务指标**：TPS、消息延迟、在线会话数、GC 次数/时间、内存使用、错误率等
- **使用示例**：
  ```csharp
  MetricsCollector.Instance.RegisterCounter("server.rps");
  MetricsCollector.Instance.IncrementCounter("server.rps", 1);
  var snapshot = MetricsCollector.Instance.GetCounterSnapshot();
  ```

### 8.2 健康检查 (Health Check)

- **可插拔检查项**：支持注册自定义健康检查项（TCP 监听、会话数、内存使用等）
- **聚合报告**：HealthCheckService 聚合所有检查项结果，输出 JSON 或纯文本报告
- **三种状态**：Healthy（健康）、Degraded（降级）、Unhealthy（异常）
- **使用示例**：
  ```csharp
  HealthCheckService.Instance.RegisterCheck("tcp", () => 
      tcpServer.IsRunning ? HealthCheckResult.Healthy("TCP 监听正常") 
                         : HealthCheckResult.Unhealthy("TCP 监听异常"));
  var report = HealthCheckService.Instance.GetReportAsJson();
  ```

### 8.3 消息限流 (Rate Limiting)

- **令牌桶算法**：固定速率补充令牌，支持流量突发
- **两级限流**：会话级（单玩家速率限制）和全局级（总消息速率限制）
- **防刷攻击**：异常流量自动丢弃，保护服务器核心资源
- **使用示例**：
  ```csharp
  var sessionLimiter = new RateLimiter(100, 5000); // 100 TPS，5000 令牌/秒
  var globalLimiter = new RateLimiter(10000, 500000);
  if (globalLimiter.TryConsume()) { /* 处理消息 */ }
  ```

### 8.4 服务状态管理 (Service State)

- **状态机流转**：Stopped → Starting → Running → Pausing → Paused → Stopping → Stopped
- **事件订阅**：StateChanged 事件可订阅，用于状态变更时的通知和联动
- **使用示例**：
  ```csharp
  ServiceStateManager.Instance.StateChanged += (s, e) => 
      Logger.Info("Server", $"状态变更: {e.Previous} -> {e.Current}");
  ServiceStateManager.Instance.SetRunning();
  ```

### 8.5 崩溃收集 (Crash Reporting)

- **未处理异常捕获**：自动订阅 AppDomain.UnhandledException 和 TaskScheduler.UnobservedTaskException
- **结构化崩溃报告**：包含时间戳、异常类型、堆栈跟踪、服务状态、内存/GC 信息
- **文件落盘**：自动写入 logs/crashes/ 目录，便于事后分析

### 8.6 可插拔配置 (FrameworkOptions)

- **统一配置入口**：FrameworkOptions 类集中管理所有模块的启用状态
- **生产/调试预设**：FrameworkOptions.Production() 和 FrameworkOptions.Debug() 一键配置
- **统一初始化**：FrameworkBootstrap 类提供一次性初始化框架所有模块的入口
- **使用示例**：
  ```csharp
  var options = FrameworkOptions.Production()
      .WithMetrics()
      .WithHealthCheck()
      .WithRateLimiting()
      .WithServiceState()
      .WithCrashReporting();
  FrameworkBootstrap.Initialize(options, logger);
  ```
