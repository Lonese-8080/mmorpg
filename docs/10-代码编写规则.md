## 1. 核心原则

### 1.1 黄金法则：可读性第一

```
┌─────────────────────────────────────────────────────────────────────┐
│  🎯 黄金法则：代码首先是写给人看的，顺便让机器执行                   │
│                                                                     │
│  ✅ 正确的态度：                                                  │
│  "写代码时，想象三个月后的自己正在调试这段代码。                      │
│   那时的你，会对现在的你说什么？"                                    │
│                                                                     │
│  ✅ 正确的优先级：                                                 │
│  1. 可读（让人能看懂）                                             │
│  2. 可维护（让人能修改）                                           │
│  3. 可测试（让人能测试）                                           │
│  4. 性能（让机器跑得快）                                           │
│                                                                     │
│  ❌ 错误的态度：                                                  │
│  "能运行就行"、"先写出来再说"、"别人会看懂的"                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 KISS 原则

```
KISS = Keep It Simple, Stupid（保持简单，笨蛋）

✅ 正确示例：一个方法只做一件事，清晰明了。
❌ 错误示例：一个方法做五件事，方法签名 30 个参数。

判断标准：如果你无法用一句话描述这个方法做什么，它就应该被拆分。
```

### 1.3 SOLID 原则

```
本项目严格遵循 SOLID 原则：

┌─────────────────────────────────────────────────────────────┐
│  S - 单一职责原则 (Single Responsibility)                    │
│      → 一个类只做一件事                                       │
│                                                               │
│  O - 开闭原则 (Open-Closed)                                  │
│      → 对扩展开放，对修改关闭                                  │
│                                                               │
│  L - 里氏替换原则 (Liskov Substitution)                      │
│      → 子类可以替换父类使用                                   │
│                                                               │
│  I - 接口隔离原则 (Interface Segregation)                    │
│      → 多个小接口优于一个大接口                               │
│                                                               │
│  D - 依赖倒置原则 (Dependency Inversion)                     │
│      → 依赖抽象，不依赖具体                                   │
└─────────────────────────────────────────────────────────────┘
```

### 1.4 四不要原则

```
┌─────────────────────────────────────────────────────────────────┐
│  ❌ 不要编写"聪明"的代码                                        │
│      示例：使用晦涩的位运算、奇怪的语法、过度的 LINQ              │
│      后果：只有你自己能维护，三个月后你也不懂了                  │
│                                                                 │
│  ❌ 不要提前优化                                                │
│      示例：为不存在的性能问题编写复杂的缓存系统                  │
│      原则：先让它运行正确，再让它运行快                           │
│                                                                 │
│  ❌ 不要魔法值                                                  │
│      示例：if (level == 85) ... // 85 是什么意思？              │
│      正确：const int MaxPlayerLevel = 85;                       │
│                                                                 │
│  ❌ 不要沉默失败                                                │
│      示例：catch (Exception) { } // 吞掉异常                    │
│      正确：要么处理，要么向上抛，要么记录日志                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

```csharp
// ====================================================================
// 作者: 伙伴团队
// 说明: 玩家管理器 - 负责管理所有在线玩家（示例类，当前代码库中不存在）
// ====================================================================

using System;
using System.Collections.Generic;
using System.Threading;
using System.Threading.Tasks;

using MMORPG.Framework;
using MMORPG.Core.ECS;

namespace MMORPG.Game
{
    /// <summary>
    /// 玩家管理器 - 负责管理所有在线玩家
    /// 
    /// 职责：
    ///   1. 维护在线玩家集合
    ///   2. 提供玩家查找接口
    ///   3. 处理玩家上线/下线事件
    /// 
    /// 线程安全性：线程安全（内部使用锁保护）
    /// </summary>
    public class PlayerManager
    {
        // ============================================================
        // 字段区（按功能分组，使用 #region 组织）
        // ============================================================
        
        /// <summary>
        /// 玩家 ID 到玩家对象的映射字典
        /// Key：玩家唯一标识 ID
        /// Value：玩家实体对象
        /// </summary>
        private readonly Dictionary<long, PlayerEntity> _playersById;
        
        /// <summary>
        /// 会话 ID 到玩家 ID 的映射
        /// 用于快速查找"哪个会话对应哪个玩家"
        /// </summary>
        private readonly Dictionary<long, long> _sessionToPlayerMap;
        
        // ============================================================
        // 构造函数区
        // ============================================================
        
        /// <summary>
        /// 初始化玩家管理器
        /// </summary>
        public PlayerManager()
        {
            _playersById = new Dictionary<long, PlayerEntity>();
            _sessionToPlayerMap = new Dictionary<long, long>();
        }
        
        // ============================================================
        // 方法区（按功能分组，使用 #region 组织）
        // ============================================================
        
        #region 玩家上线/下线
        
        /// <summary>
        /// 玩家上线
        /// </summary>
        /// <param name="playerId">玩家 ID</param>
        /// <param name="player">玩家实体</param>
        /// <returns>是否添加成功（true = 成功，false = 玩家已在线）</returns>
        public bool AddPlayer(long playerId, PlayerEntity player)
        {
            // 防止重复添加
            if (_playersById.ContainsKey(playerId))
            {
                Logger.Warning("PlayerManager", "玩家 {0} 已在线，拒绝重复添加", playerId);
                return false;
            }
            
            _playersById[playerId] = player;
            
            Logger.Info("PlayerManager", "玩家 {0} 上线，当前在线 {1} 人",
                playerId, _playersById.Count);
            
            return true;
        }
        
        /// <summary>
        /// 玩家下线
        /// </summary>
        /// <param name="playerId">玩家 ID</param>
        /// <returns>是否成功移除（true = 成功，false = 玩家不存在）</returns>
        public bool RemovePlayer(long playerId)
        {
            if (_playersById.Remove(playerId))
            {
                Logger.Info("PlayerManager", "玩家 {0} 下线，当前在线 {1} 人",
                    playerId, _playersById.Count);
                return true;
            }
            
            Logger.Debug("PlayerManager", "玩家 {0} 不存在，无法下线", playerId);
            return false;
        }
        
        #endregion
        
        #region 查询接口
        
        /// <summary>
        /// 根据玩家 ID 查找玩家实体
        /// </summary>
        /// <param name="playerId">玩家 ID</param>
        /// <returns>玩家实体（未找到返回 null）</returns>
        public PlayerEntity? FindById(long playerId)
        {
            if (_playersById.TryGetValue(playerId, out var player))
            {
                return player;
            }
            
            return null;
        }
        
        /// <summary>
        /// 获取在线玩家总数
        /// </summary>
        /// <returns>在线玩家数量</returns>
        public int GetOnlineCount()
        {
            return _playersById.Count;
        }
        
        #endregion
    }
}
```

### 3.2 命名的黄金实践

```
✅ 命名要精确，不要模糊
   ❌ ProcessMessage()   → 处理什么消息？
   ✅ ParseLoginMessage() → 解析登录消息

✅ 使用行业术语
   ❌ UserLogin()      → 什么 User？
   ✅ PlayerLogin()     → 玩家登录（游戏行业术语）

✅ 方法名应该描述返回值
   ❌ GetPlayer()      → 找不到会怎样？
   ✅ FindPlayerById() → 很明确：按 ID 查找，可能找不到
   ✅ GetPlayerByIdOrThrow() → 找不到会抛出异常

✅ 使用对称命名
   ✅ Open() / Close()
   ✅ Connect() / Disconnect()
   ✅ Start() / Stop()
   ✅ Add() / Remove()
   ✅ Create() / Delete()

✅ 布尔值的好名字
   ❌ flag, ok, result, status
   ✅ isInitialized, hasValue, canConnect, shouldProcess

✅ 集合变量名
   ❌ players → 不明确是单个还是集合
   ✅ playerList, playerArray, playerDictionary
   ✅ playersBySessionId（按会话 ID 的玩家字典）
```

```

│  ❌ 无用的注释（只描述代码表面含义）：                           │

│  // 打印输出                                                    │
│  // 循环 i 从 0 到 10                                          │
│                                                                 │
│  注释应该回答的问题：                                           │
│  1. 为什么这样写？                                             │
│  2. 为什么不那样写？                                           │
│  3. 这里有什么坑？                                             │
│  4. 注意事项是什么？                                          │
│  5. 性能/安全方面有什么需要注意的？                            │

```

### 4.2 方法注释示例

```csharp
/// <summary>
/// 发送消息给指定玩家
/// 
/// 功能：
///   1. 查找玩家对应的会话
///   2. 将消息序列化后发送给玩家
///   3. 记录发送日志
/// 
/// 返回值：
///   true  - 消息已成功放入发送队列
///   false - 玩家不在线或会话已断开
/// 
/// 注意事项：
///   - 这是异步方法，消息被放入队列后立即返回
///   - 不保证消息一定被送达（网络可能断开）
///   - 如果需要"必须送达"的保证，使用 SendReliableMessageAsync
/// 
/// 线程安全性：线程安全，内部使用锁保护
/// </summary>
/// <param name="playerId">玩家 ID，必须大于 0</param>
/// <param name="message">要发送的消息对象，不能为 null</param>
/// <returns>消息是否已放入发送队列</returns>
/// <exception cref="ArgumentNullException">message 参数为 null 时抛出</exception>
/// <exception cref="ArgumentOutOfRangeException">playerId <= 0 时抛出</exception>
public async Task<bool> SendMessageToPlayerAsync(long playerId, Message message)
{
    // 验证参数
    if (playerId <= 0)
    {
        throw new ArgumentOutOfRangeException(nameof(playerId), "玩家ID必须大于0");
    }
    
    if (message == null)
    {
        throw new ArgumentNullException(nameof(message));
    }
    
    // 查找玩家会话
    if (!_sessions.TryGetValue(playerId, out var session))
    {
        Logger.Debug("Network", "玩家 {0} 未找到会话，无法发送消息", playerId);
        return false;
    }
    
    // 发送消息（使用锁保护，防止多线程问题）
    using (await _lock.AcquireLock())
    {
        return await session.SendAsync(message);
    }
}
```

### 4.3 复杂算法注释示例

```csharp
/// <summary>
/// 计算玩家到目标的路径
/// 
/// 算法说明：
///   使用 A* (A-Star) 寻路算法，具体规则如下：
///   
///   1. 从起点开始，每次选择"预估距离"最小的节点
///   2. 预估距离 = 已走距离 + 到目标的曼哈顿距离
///   3. 已走距离用真实路径长度
///   4. 障碍物不可通过
/// 
/// 时间复杂度：O(n log n)，n 为地图节点数
/// 空间复杂度：O(n)，需要存储已访问和待访问的节点
/// 
/// 注意：
///   - 这是一个同步方法，调用方要注意性能影响
///   - 在大地图上可能耗时较长，建议在后台线程中调用
///   - 如果找不到路径，返回空列表
/// 
/// 性能优化：
///   - 使用最小堆（MinHeap）实现开放列表
///   - 使用 HashSet 记录已访问节点（O(1) 查询）
/// </summary>
/// <param name="start">起点坐标</param>
```

### 7.1 异常处理原则

## 8. 性能优化规范

### 8.1 性能原则

```
┌─────────────────────────────────────────────────────────────┐
│  🎯 性能原则：先让它运行正确，再让它运行快                      │
│                                                                 │
│  1. 不要过早优化 (Premature Optimization)                      │
│     先写清晰的代码，再通过性能测试识别瓶颈                        │
│                                                                 │
│  2. 用数据说话                                                 │
│     不要"我觉得这里慢"，用 profiling 工具找到真正的热点          │
│                                                                 │
│  3. 80-20 法则                                                │
│     80% 的执行时间花在 20% 的代码上，优化那 20% 就够了         │
│                                                                 │
│  4. 内存分配比你想象的贵                                        │
│     GC 停顿是服务器性能的大敌，尽量减少对象分配                 │
│                                                                 │
│  5. 缓存是银弹，但也是毒药                                     │
│     缓存能让程序变快，但也带来一致性、内存、过期等问题           │
│     除非确定需要，否则不要一开始就设计缓存                       │
│                                                                 │
└─────────────────────────────────────────────────────────────┘
```

```csharp
public class MessagePool
```

```csharp
/// <summary>
/// 验证玩家移动消息
/// 
/// 验证内容：
///   1. 玩家必须存在
///   2. 目标坐标必须在合法范围内
///   3. 移动距离不能超过最大移动速度（防瞬移挂）
///   4. 不能穿墙
/// </summary>
public async Task HandleMoveAsync(Session session, MoveMessage message)
{
    // 1. 玩家必须存在且已登录
    if (session.PlayerId <= 0)
    {
        Logger.Warning("Game", "未登录的玩家尝试移动: {0}", session.Id);
        await session.SendErrorAsync(ErrorCode.NotLoggedIn, "请先登录");
        return;
    }
    
    // 2. 坐标格式和范围
    const float MinCoord = -10000.0f;
    const float MaxCoord = 10000.0f;
    
    if (!IsValidFloat(message.TargetX) ||
        !IsValidFloat(message.TargetY) ||
        !IsValidFloat(message.TargetZ))
    {
        Logger.Warning("Game", "收到非法的坐标值: {0}, {1}, {2}",
            message.TargetX, message.TargetY, message.TargetZ);
        await session.DisconnectAsync("非法的坐标数据");
        return;
    }
    
    // 3. 移动距离（防瞬移挂）
    var currentPos = GetPlayerPosition(session.PlayerId);
    var distance = CalculateDistance(currentPos, 
        new Vector3(message.TargetX, message.TargetY, message.TargetZ));
    
    const float MaxMoveDistance = 10.0f; // 10 米
    
    if (distance > MaxMoveDistance)
    {
        Logger.Warning("Game", "玩家移动距离过大: {0} -> 距离 {1} 米",
            session.PlayerId, distance);
        
        await session.DisconnectAsync("疑似使用外挂");
        return;
    }
    
    // 4. 碰撞检测（不能穿墙）
    if (IsPathBlocked(currentPos, message.TargetX, message.TargetY, message.TargetZ))
    {
        Logger.Warning("Game", "玩家尝试穿墙: {0}", session.PlayerId);
        await session.SendErrorAsync(ErrorCode.InvalidPosition, "路径被阻挡");
        return;
    }
    
    // 所有验证通过，执行真正的逻辑
    MovePlayer(session.PlayerId, message.TargetX, message.TargetY, message.TargetZ);
    
    // 广播给周围玩家
    BroadcastPositionToNearbyPlayers(session.PlayerId, 
        message.TargetX, message.TargetY, message.TargetZ);
}

/// <summary>

/// </summary>
```

```
│  🎯 标准 TDD 流程（红 → 绿 → 重构）                           │
│                                                                 │
│  1. RED（红色）：编写测试（测试会失败，因为功能还没实现）         │
│  2. GREEN（绿色）：实现功能（让测试通过）                      │
│  3. REFACTOR（重构）：优化代码（保持测试通过）                  │
│                                                                 │
│  简化流程（对初学者）：                                         │
│  1. 先实现功能                                                 │
│  2. 立即编写测试（验证功能正确）                                │
│  3. 运行测试，确保通过                                          │
│  4. 修改功能，确保测试仍然通过                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────┘
```

### 10.2 测试类型和覆盖要求

```
┌─────────────────────────────────────────────────────────────────┐
│  测试类型         │  覆盖要求         │  目标                 │
├──────────────────┼──────────────┼───────────────────────┤
│  单元测试        │  ✅ 必须    │  所有业务逻辑，每个方法   │
│  集成测试        │  ✅ 必须    │  关键流程           │
│  性能测试        │  ✅ 关键模块  │  性能关键路径         │
│  回归测试        │  ✅ 关键模块  │  每个 Bug 修复         │
│  边界测试        │  ✅ 所有方法  │  边界条件         │
└──────────────────┴──────────────┴───────────────────────┘
```

### 10.3 单元测试示例

```csharp
namespace MMORPG.Game.Tests
{
    /// <summary>
    /// 玩家管理器测试
    /// 
    /// 测试目标：
    /// 1. 玩家上线/下线逻辑
    /// 2. 查询接口正确性
    /// 3. 边界条件测试
    /// </summary>
    // 注意：PlayerManagerTests 为测试规范示例，当前代码库中不存在此类
    public class PlayerManagerTests
    {
        /// <summary>
        /// 测试：添加新玩家上线
        /// </summary>
        [Test]
        public void AddPlayer_NewPlayer_AddsSuccessfully()
        {
            // Arrange（准备）
            var manager = new PlayerManager();
            var player = new PlayerEntity { Id = 12345, Name = "测试玩家" };
            
            // Act（执行）
            var result = manager.AddPlayer(12345, player);
            
            // Assert（验证）
            Assert.True(result);
            Assert.Equal(1, manager.GetOnlineCount());
        }
        
        /// <summary>
        /// 测试：重复添加同一玩家
        /// 期望：第二次添加应返回 false
        /// </summary>
        [Test]
        public void AddPlayer_Duplicate_ReturnsFalse()
        {
            // Arrange
            var manager = new PlayerManager();
            var player = new PlayerEntity { Id = 12345, Name = "测试玩家" };
            
            // Act
            manager.AddPlayer(12345, player);
            var result = manager.AddPlayer(12345, player);
            
            // Assert
            Assert.False(result);
            Assert.Equal(1, manager.GetOnlineCount());
        }
        
        /// <summary>
        /// 测试：查找存在的玩家
        /// </summary>
        [Test]
        public void FindById_ExistingPlayer_ReturnsPlayer()
        {
            // Arrange
            var manager = new PlayerManager();
            var player = new PlayerEntity { Id = 12345, Name = "测试玩家" };
            manager.AddPlayer(12345, player);
            
            // Act
            var result = manager.FindById(12345);
            
            // Assert
            Assert.NotNull(result);
            Assert.Equal("测试玩家", result.Name);
        }
        
        /// <summary>
        /// 测试：查找不存在的玩家
        /// 期望：返回 null
        /// </summary>
        [Test]
        public void FindById_NonExisting_ReturnsNull()
        {
            // Arrange
            var manager = new PlayerManager();
            
            // Act
            var result = manager.FindById(99999);
            
            // Assert
            Assert.Null(result);
        }
    }
}
```

## 14. 文档维护规范

### 14.1 文档更新规则

```
┌─────────────────────────────────────────────────────────────┐
│  📌 规则1：代码和文档一起提交                              │
│                                                                 │
│  ✅ 修改代码时，同时检查并更新相关文档                          │
│  ✅ 新增功能时，编写或更新相应文档                        │
│  ✅ 修改协议时，同步更新协议文档                          │
│  ✅ 修改配置时，同步更新配置文档                          │
│                                                                 │
│  ❌ 只改代码不改文档 → 其他人看不懂你的代码                  │
│  ❌ 先写代码后补文档 → 永远不会写                   │
│  ❌ 文档和代码分开提交 → 容易忘记                           │
│                                                                 │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  📌 规则2：文档和代码同等重要                                │
│                                                                 │
│  文档的价值：                                                  │
│  1. 新成员快速理解系统设计意图                               │
│  2. 你自己三个月后回头看                                        │
│  3. 排错时快速定位问题                                       │
│  4. 新人接手时的救命稻草                                       │
│                                                                 │
│  代码质量 = 代码可读性 + 注释质量 + 文档质量                           │
│                                                                 │
└─────────────────────────────────────────────────────────────┘
```

### 12.2 文档更新时机

```
✅ 必须更新文档的情况：

1. 新增功能/模块
   → 编写新的设计文档或更新 API 文档

2. 修改核心逻辑
   → 更新相应的设计文档

3. 修改公开 API
   → 更新 API 参考文档

4. 添加新的消息协议
   → 更新协议设计文档

5. 修改配置格式
   → 更新配置文档

6. 修复 Bug
   → 在变更日志中记录

7. 性能优化
   → 更新性能文档或添加性能文档

8. 修改架构/设计
   → 更新架构文档
```

## 15. 违反规范的处理

### 15.1 违反级别

```
┌─────────────────────────────────────────────────────────────────┐
│  级别         │  定义                                         │
├──────────────┼───────────────────────────────────────────────┤
│  Level 1 轻微 │  命名不规范、注释缺失、小格式问题           │
│              │  处理：代码审查时提出，下次提交前修复                │
│              │                                               │
│  Level 2 一般 │  缺少测试、缺少注释、可读性差                 │
│              │  处理：必须在合并前修复                          │
│              │                                               │
│  Level 3 严重 │  安全隐患、重大性能问题、异常吞掉异常           │
│              │  处理：必须立即修复，不能合并               │
│              │                                               │
│  Level 4 致命 │  代码无法运行、功能错误、数据丢失风险         │
│              │  处理：绝对不能合并，必须修复后重新审查            │
└──────────────┴───────────────────────────────────────────────┘
```
