## 1. 设计目标

### 1.1 为什么需要序列化？

```
客户端 ──────▶ 服务器

问题：
1. 客户端用 C#/Unity/C++ 写的
2. 服务器用 C# 写的
3. 数据怎么传递？

解决方案：序列化

客户端（发送）：
  玩家移动 = {X: 100, Y: 200, Z: 0}
      ↓ 序列化（Protobuf）
  二进制数据 = [08 64 10 C8 01 18 00]
      ↓ 发送
  ─────────────────────────────────────────────▶
      ↓ 接收
  二进制数据 = [08 64 10 C8 01 18 00]
      ↓ 反序列化（Protobuf）
  玩家移动 = {X: 100, Y: 200, Z: 0}
  服务器（接收）
```

### 1.2 性能要求

```
┌─────────────────────────────────────────────────────────────┐
│                      序列化性能指标                            │
├─────────────────────┬────────────────────────────────────────┤
│       指标           │              目标值                     │
├─────────────────────┼────────────────────────────────────────┤
│ 序列化速度           │         > 100MB/s                      │
├─────────────────────┼────────────────────────────────────────┤
│ 反序列化速度         │         > 100MB/s                      │
├─────────────────────┼────────────────────────────────────────┤
│ 序列化后体积         │         JSON 的 1/3 ~ 1/10            │
├─────────────────────┼────────────────────────────────────────┤
│ GC 分配（每消息）    │         < 50 bytes                     │
└─────────────────────┴────────────────────────────────────────┘
```

## 2. 技术选型

### 2.1 方案对比

```
┌─────────────────────────────────────────────────────────────────┐
│                      序列化方案对比                                │
│                                                                 │
│   方案         │  速度  │  体积  │  跨语言  │  工具支持  │  GC   │
│ ───────────────┼────────┼────────┼──────────┼────────────┼───────│
│   Protobuf     │  ★★★★★ │  ★★★★★ │  ★★★★★   │  ★★★★★    │  ★★☆ │
│                                                                 │
│   MessagePack  │  ★★★★★ │  ★★★★☆ │  ★★★★☆   │  ★★★★☆    │  ★★★ │
│                                                                 │
│   MemoryPack   │  ★★★★★ │  ★★★★★ │  ★☆☆☆☆   │  ★★☆☆    │  ★★★★★│
│   (.NET专用)   │        │        │          │            │       │
│                                                                 │
│   System.Text  │  ★★★☆☆ │  ★★☆☆☆ │  ★★★★☆   │  ★★★★★    │  ★☆☆ │
│   .Json        │        │        │          │            │       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

最终选择：Protobuf
原因：
1. 跨语言（Unity C# + 服务器 C# + 未来可能的 Go/Java）
2. 成熟稳定（Google 出品）
3. 工具完善（protoc 编译器）
4. 体积小（关键！网络带宽有限）
5. 向后兼容（增删字段不影响旧客户端）
```

### 2.2 为什么不用 MessagePack？

```
MessagePack 的问题：
1. 字段名丢失，只靠顺序
   - 修改字段顺序会破坏兼容性
   - 删除字段也危险
   
2. 数字没有类型
   - int 可能是 1 字节、2 字节、4 字节...
   - 解析时需要猜

3. 不支持默认值
   - 每个字段都要传，即使没变
   - 浪费带宽

Protobuf 的优势：
1. 有字段名和序号
   - 修改字段顺序不影响
   - 删除字段可以标记 deprecated
   
2. 类型明确
   - int32, int64, uint32, float, double...
   - 解析精确

3. 支持默认值
   - int 默认 0，string 默认 ""
   - 传输时可以省略

4. proto3 简洁语法
   - 无需 required/optional 标记
   - 生成的代码更小
```

```protobuf
// common.proto
// 框架层通用消息定义
// 所有游戏都会用到的基本消息

syntax = "proto3";

package MMORPG.Common;

option csharp_namespace = "MMORPG.Framework.Network";

// ============================================================
// 框架层消息 ID: 0x0000 - 0x0FFF
// ============================================================

// ----------------------------------------
// 登录相关 (0x0001 - 0x001F)
// ----------------------------------------

/// 客户端登录请求
message C2S_Login
{
    /// 账号
    string account = 1;
    
    /// 密码（加盐哈希后）
    string password = 2;
    
    /// 客户端版本
    string client_version = 3;
    
    /// 平台（iOS/Android/PC）
    string platform = 4;
}

/// 服务器登录响应
message S2C_LoginResult
{
    /// 是否成功
    bool success = 1;
    
    /// 错误码（success=false 时填充）
    int32 error_code = 2;
    
    /// 错误消息
    string error_message = 3;
    
    /// 玩家ID（success=true 时填充）
    int64 player_id = 4;
    
    /// 认证Token
    string token = 5;
    
    /// 服务器时间戳
    int64 server_time = 6;
}

// ----------------------------------------
// 心跳相关 (0x0020 - 0x002F)
// ----------------------------------------

/// 客户端心跳
message C2S_Heartbeat
{
    /// 客户端本地时间戳（毫秒）
    int64 client_time = 1;
}

/// 服务器心跳响应
message S2C_Heartbeat
{
    /// 回显的客户端时间戳（用于计算 RTT）
    int64 client_time = 1;
    
    /// 服务器当前时间戳
    int64 server_time = 2;
}

// ----------------------------------------
// 进入世界 (0x0030 - 0x003F)
// ----------------------------------------

/// 客户端请求进入游戏世界
message C2S_EnterWorld
{
    /// 玩家ID
    int64 player_id = 1;
    
    /// 角色名称
    string character_name = 2;
}

/// 服务器响应进入游戏世界
message S2C_EnterWorld
{
    /// 是否成功
    bool success = 1;
    
    /// 玩家ID
    int64 player_id = 2;
    
    /// 出生坐标X
    float position_x = 3;
    
    /// 出生坐标Y
    float position_y = 4;
    
    /// 出生坐标Z
    float position_z = 5;
    
    /// 失败时的错误原因
    string error_message = 6;
}

// ----------------------------------------
// 服务器通知 (0x0100 - 0x01FF)
// ----------------------------------------

/// 服务器公告
message S2C_ServerNotice
{
    /// 公告内容
    string notice = 1;
}

/// 服务器错误
message S2C_Error
{
    /// 错误代码
    int32 error_code = 1;
    
    /// 错误消息
    string message = 2;
}

// ----------------------------------------
// 移动相关 (0x1000 - 0x10FF)
// ----------------------------------------

/// 客户端移动请求
message C2S_PlayerMove
{
    /// 目标X
    float x = 1;
    
    /// 目标Y
    float y = 2;
    
    /// 目标Z
    float z = 3;
    
    /// 朝向角度（0-360）
    float yaw = 4;
    
    /// 移动类型：0=停止 1=行走 2=跑步 3=跳跃
    int32 move_type = 5;
}

/// 服务器广播玩家位置变化
message S2C_PlayerPosition
{
    /// 玩家ID
    int64 player_id = 1;
    
    /// 新坐标X
    float x = 2;
    
    /// 新坐标Y

    /// 新坐标Z
    float z = 4;
    
    /// 朝向角度

}
```

## 5. 序列化实现

### 5.1 TCP 消息包格式

```
┌─────────────────────────────────────────────────────────────────┐
│                      TCP 消息包格式                              │
│                                                                 │
│  ┌──────────────┬──────────────┬────────────────────────────┐  │
│  │ 消息长度(4B) │ 消息ID(4B)  │   Protobuf 消息体(N字节)    │  │
│  └──────────────┴──────────────┴────────────────────────────┘  │
│                                                                 │
│  总长度 = 8 + 消息体长度                                         │
│                                                                 │
│  注意：                                                         │
│  - 所有数字字段使用小端序（BitConverter 默认为小端）             │
│  - 最大消息长度限制为 1MB（防止恶意客户端攻击）                   │
│  - 消息体由 Protobuf 序列化生成                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 MessageSerializer 实现

```csharp
namespace MMORPG.Framework.Network;

using Google.Protobuf;
using System.Buffers;

/// <summary>
/// 消息序列化器（Protobuf 实现）
///
/// TCP 消息包格式：
/// ┌───────────────────────────────────────────────────────────────┐
/// │  消息体长度(4字节 int) │  消息ID(4字节 uint) │  Protobuf消息体(N字节) │
/// └───────────────────────────────────────────────────────────────┘
///
/// 消息体由 Google.Protobuf IMessage.WriteTo() 序列化
/// </summary>
public static class MessageSerializer
{
    /// <summary>
    /// 消息头大小（长度4 + 类型4 = 8字节）
    /// </summary>
    public const int HeaderSize = 8;

    /// <summary>
    /// 最大消息体大小（1MB）
    /// </summary>
    public const int MaxBodySize = 1024 * 1024;

    /// <summary>
    /// 消息工厂：根据消息 ID 创建消息实例
    /// </summary>
    private static readonly Dictionary<uint, Func<IMessage>> _messageFactory = new()
    {
        [MessageIds.C2S_Login]        = () => new C2S_Login(),
        [MessageIds.S2C_LoginResult]  = () => new S2C_LoginResult(),
        [MessageIds.C2S_Heartbeat]    = () => new C2S_Heartbeat(),
        [MessageIds.S2C_Heartbeat]    = () => new S2C_Heartbeat(),
        [MessageIds.C2S_EnterWorld]   = () => new C2S_EnterWorld(),
        [MessageIds.S2C_EnterWorld]   = () => new S2C_EnterWorld(),
        [MessageIds.S2C_ServerNotice] = () => new S2C_ServerNotice(),
        [MessageIds.S2C_Error]        = () => new S2C_Error(),
        [MessageIds.C2S_PlayerMove]   = () => new C2S_PlayerMove(),
        [MessageIds.S2C_PlayerPosition] = () => new S2C_PlayerPosition(),
    };

    /// <summary>
    /// 注册自定义消息类型
    /// </summary>
    public static void Register<T>(uint messageId) where T : IMessage, new()
    {
        _messageFactory[messageId] = () => new T();
    }

    /// <summary>
    /// 序列化消息到池化缓冲区（零分配版本）
    /// </summary>
    public static int SerializeToPooledBuffer(IMessage message, out byte[] buffer)
    {
        // 1. 计算 Protobuf 消息体长度
        var bodyLength = message.CalculateSize();

        // 2. 验证消息体长度
        if (bodyLength > MaxBodySize)
        {
            throw new InvalidOperationException($"消息体长度 {bodyLength} 超过最大限制 {MaxBodySize}");
        }

        // 3. 从 ArrayPool 租用缓冲区
        var totalSize = HeaderSize + bodyLength;
        buffer = ArrayPool<byte>.Shared.Rent(totalSize);

        // 4. 写入消息头
        // 消息体长度（4字节，小端序）
        buffer[0] = (byte)bodyLength;
        buffer[1] = (byte)(bodyLength >> 8);
        buffer[2] = (byte)(bodyLength >> 16);
        buffer[3] = (byte)(bodyLength >> 24);
        // 消息类型 ID（4字节，小端序）
        var messageId = GetMessageId(message);
        buffer[4] = (byte)messageId;
        buffer[5] = (byte)(messageId >> 8);
        buffer[6] = (byte)(messageId >> 16);
        buffer[7] = (byte)(messageId >> 24);

        // 5. 写入 Protobuf 消息体
        message.WriteTo(buffer.AsSpan(HeaderSize));

        return totalSize;
    }

    /// <summary>
    /// 归还池化缓冲区
    /// </summary>
    public static void ReturnBuffer(byte[] buffer)
    {
        ArrayPool<byte>.Shared.Return(buffer);
    }

    /// <summary>
    /// 序列化消息为完整的 TCP 数据包
    /// </summary>
    public static byte[] Serialize(IMessage message)
    {
        var pooled = SerializeToPooledBuffer(message, out var pooledBuffer);
        try
        {
            var result = new byte[pooled];
            Buffer.BlockCopy(pooledBuffer, 0, result, 0, pooled);
            return result;
        }
        finally
        {
            ArrayPool<byte>.Shared.Return(pooledBuffer);
        }
    }

    /// <summary>
    /// 从字节数组反序列化消息
    /// </summary>
    public static IMessage Deserialize(byte[] data, int offset = 0)
    {
        // 1. 读取消息体长度
        var bodyLength = BitConverter.ToInt32(data, offset);

        // 2. 验证长度
        if (bodyLength < 0 || bodyLength > MaxBodySize)
        {
            throw new InvalidOperationException($"非法的消息体长度: {bodyLength}");
        }

        // 3. 读取消息类型
        var messageId = BitConverter.ToUInt32(data, offset + 4);

        // 4. 创建消息实例
        if (!_messageFactory.TryGetValue(messageId, out var factory))
        {
            throw new InvalidOperationException($"未注册的消息类型: 0x{messageId:X8}");
        }

        var message = factory();

        // 5. 使用 Protobuf 解析消息体
        message.MergeFrom(data, offset + HeaderSize, bodyLength);

        return message;
    }

    /// <summary>
    /// 从字节数组解析消息头（仅解析长度和类型）
    /// </summary>
    public static bool TryParseHeader(byte[] data, int offset, out int bodyLength, out uint messageId)
    {
        bodyLength = 0;
        messageId = 0;

        if (data == null || data.Length - offset < HeaderSize)
            return false;

        bodyLength = BitConverter.ToInt32(data, offset);
        messageId = BitConverter.ToUInt32(data, offset + 4);

        if (bodyLength < 0 || bodyLength > MaxBodySize)
            return false;

        return true;
    }

    /// <summary>
    /// 获取消息的消息 ID
    /// </summary>
    private static uint GetMessageId(IMessage message)
    {
        // 根据消息类型返回对应的 ID
        return message switch
        {
            C2S_Login => MessageIds.C2S_Login,
            S2C_LoginResult => MessageIds.S2C_LoginResult,
            C2S_Heartbeat => MessageIds.C2S_Heartbeat,
            S2C_Heartbeat => MessageIds.S2C_Heartbeat,
            C2S_EnterWorld => MessageIds.C2S_EnterWorld,
            S2C_EnterWorld => MessageIds.S2C_EnterWorld,
            S2C_ServerNotice => MessageIds.S2C_ServerNotice,
            S2C_Error => MessageIds.S2C_Error,
            C2S_PlayerMove => MessageIds.C2S_PlayerMove,
            S2C_PlayerPosition => MessageIds.S2C_PlayerPosition,
            _ => throw new InvalidOperationException($"未知消息类型: {message.GetType().Name}")
        };
    }
}
```

## 7. 高级话题

### 7.1 生成 Protobuf 代码

**使用 protoc 手动生成**：

```bash
# Windows
protoc --csharp_out=protos/generated protos/common.proto
protoc --csharp_out=protos/generated protos/game.proto

# Linux/macOS
protoc --csharp_out=protos/generated protos/common.proto
protoc --csharp_out=protos/generated protos/game.proto
```

**使用 MSBuild 自动生成**（推荐）：

在 `.csproj` 中添加：

```xml
<ItemGroup>
  <Protobuf Include="protos\*.proto" OutputDir="protos\generated" />
</ItemGroup>
```

需要安装 `Google.Protobuf` 和 `Grpc.Tools` NuGet 包。

### 7.2 协议演进最佳实践

```
┌─────────────────────────────────────────────────────────────────┐
│                      Protobuf 协议演进规则                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ✅ 安全操作：                                                   │
│  ├─ 新增字段（旧客户端忽略，新客户端使用默认值）                   │
│  ├─ 字段重命名（字段序号不变）                                   │
│  └─ 新增消息类型                                                 │
│                                                                  │
│  ❌ 危险操作：                                                   │
│  ├─ 删除字段（会导致旧客户端解析失败）                           │
│  ├─ 修改字段序号（破坏兼容性）                                   │
│  ├─ 修改字段类型（int32 → string 会失败）                       │
│  └─ 修改字段名称（序号不变时安全，但代码需要同步修改）             │
│                                                                  │
│  📌 建议：                                                       │
│  ├─ 删除字段时标记 deprecated 而非删除                          │
│  ├─ 新增字段使用新的序号（不要复用已删除的序号）                  │
│  └─ 重大变更时考虑新增消息类型而非修改现有消息                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 7.3 性能优化

| 优化项 | 方法 | 效果 |
|--------|------|------|
| 零拷贝序列化 | `SerializeToPooledBuffer` + `ArrayPool` | 减少 GC 分配 50% |
| 零拷贝反序列化 | `MergeFrom(byte[], offset, length)` | 避免创建临时 byte[] |
| 预计算大小 | `CalculateSize()` | 避免多次计算 |
| 消息池化 | 复用消息对象（仅适用于高频消息） | 减少 GC 分配 |
