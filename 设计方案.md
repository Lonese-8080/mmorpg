## 第零章：先搞清楚几个概念

### 0.1 什么是"高并发"？
想象你开了一家火锅店：
- **低并发**：店里只有 10 张桌子，同时只能坐 10 个人
- **高并发**：店里有 1000 张桌子，同时能坐 1000 个人，服务员还要能快速响应每一桌的需求

我们的游戏服务器要做到：**让上千甚至上万玩家同时在线，还能流畅玩游戏。**

### 0.2 什么是"多线程"？
想象厨房里有多个厨师同时炒菜：
- **单线程**：只有 1 个厨师，做完一道菜才做下一道，慢
- **多线程**：有 10 个厨师同时炒菜，快

但游戏比较特殊——**做菜（逻辑）只用1个厨师，配菜（网络/数据库）用10个厨师**。这样既能保证菜不会炒糊（不会乱序），又能充分利用多核 CPU。

### 0.3 什么是 ECS（实体-组件-系统）？
想象你在玩《我的世界》：
- **Entity（实体）**：一个 ID 编号，比如"玩家张三"、"小怪#123"、"一棵树#456"
- **Component（组件）**：一袋数据，比如"位置信息"、"血量"、"攻击力"
- **System（系统）**：处理逻辑的机器，比如"移动系统"把所有有位置组件的实体做位置更新

**核心思想：把数据和逻辑彻底分开，让数据像超市货架一样整齐排列，逻辑像流水线一样高效运转。**

| 项目 | 选择 | 为什么这么选 |
|---|---|---|
| 编程语言 | **C# 13** | 语法简洁、生态完善、性能优秀 |
| 运行时 | **.NET 10** | 性能已追近 C++，原生 AOT 支持 |
| 开发环境 | **Visual Studio 2022 / Rider** | 全中文界面，调试体验最好 |
| 目标平台 | **Windows Server / Linux** | 服务器主流平台 |

### 2.2 关键技术栈

| 模块 | 选择 | 为什么 |
|---|---|---|
| 网络IO | **SocketAsyncEventArgs + IOCP** | Windows 平台最底层、最高性能的网络模型。玩家发送的每条消息都能最低成本处理 |
| 序列化 | **Protobuf (Google)** | 成熟稳定，跨语言支持。数据打包小、速度快 |
| 消息队列 | **System.Threading.Channels** | .NET 官方高性能异步队列 |
| 内存优化 | **ArrayPool / MemoryPool** | 反复使用内存块，避免"垃圾回收停顿"（玩家会卡的元凶之一） |
| 异步模型 | **ValueTask + async/await** | 现代 C# 标准异步写法，简洁高性能 |
| 日志 | **Serilog** | 结构化日志，排查问题神器 |

## 第三章：框架层（Framework）设计详析

### 3.1 网络层 Framework.Network

#### 核心思路："接电话 + 收快递 + 发快递"

```
玩家客户端 ──▶ [监听Socket] ──▶ [玩家会话Session]
                                               │
                                               ▼
                                       [消息处理器] ──▶ 游戏逻辑
                                               │
                                               ▼
玩家客户端 ◀── [发送队列] ◀── [玩家会话Session]
```

**关键组件：**

- **TcpServer**：门口保安，负责"接电话"，监听某个端口（比如 9000），玩家连接进来时创建一个 Session
- **Session**：每一个在线玩家对应一个 Session，代表"当前这次连接"，负责收/发该玩家的消息
- **MessageRouter**：邮局分拣员，收到消息后根据"消息类型编号"分发给对应的处理函数

**消息格式（简化版）：**

```
┌──────────────┬──────────────┬────────────────────────────┐
│  消息长度(4B) │ 消息类型(2B) │   消息内容（Protobuf二进制） │
└──────────────┴──────────────┴────────────────────────────┘
        ↑
    告诉我们这条消息有多长，
    让我们知道从哪里切开下一条
```

### 3.2 线程调度 Framework.Threading

#### 核心思路："单线程跑游戏逻辑 + 多线程跑IO"

```
┌──────────────────────────────────────────────────────┐
│                  主线程（游戏逻辑）                   │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐             │
│  │ ECS  │→ │ 场景 │→ │ 战斗 │→ │ 广播 │→ 循环...    │
│  └──────┘  └──────┘  └──────┘  └──────┘             │
│         ↑        不需要锁，因为只有一个线程           │
│         │                                            │
│   ┌──────────────────────────────────────┐          │
│   │           消息队列 (Channel)          │◀── 网络线程
│   └──────────────────────────────────────┘          │
│         │                                            │
│         ▼                                            │
│   ┌──────────────────────────────────────┐          │
│   │           发送队列 (Channel)          │──▶ 网络线程
│   └──────────────────────────────────────┘          │
└──────────────────────────────────────────────────────┘
```

**为什么游戏逻辑用单线程？**
- 想象你在记账：如果有 10 个人同时在你账本上写字，账本就乱了
- 一个人写，虽然慢点，但绝不会出错
- 我们靠"写得很快"（每帧 50ms）来弥补单线程的局限
- 网络IO、数据库操作这些"等待的事"交给其他线程做，不耽误主线程

### 3.3 序列化 Framework.Network（Protobuf）

- 使用 **Google Protobuf** 定义消息协议（`.proto` 文件）
- 每种消息 = 一个 proto message，`protoc` 自动生成 C# 代码
- 好处：
  - **跨语言**：Unity C# / 服务端 C# / 未来 Go/Java 都能用同一套协议
  - **向后兼容**：增删字段不影响旧客户端
  - **体积小**：比 JSON 小 3-10 倍，节省带宽
  - **调试方便**：`protoc --decode` 直接解析二进制报文

**协议文件位置**：`protos/common.proto`（框架层）、`protos/game.proto`（业务层）

### 3.4 日志 Framework.Logging ✅ 已实现

- 每个模块一个 Logger
- 日志级别：Verbose < Debug < Info < Warning < Error < Fatal
- 输出到：控制台 + 文件 +（可选）远程日志服务

### 3.5 定时器 Framework.Timer

- 每隔 X 毫秒自动执行某件事
- 用途：玩家心跳检测、定时刷新小怪、每日任务重置等
- 实现：优先级队列 + 主线程每帧检查到期任务

## 第四章：ECS 架构详析（自研 Archetype 模式）

### 4.1 Archetype 是什么？

想象你有一堆小朋友（实体），每个小朋友带不同的书包（组件）：

- 小朋友 A 带了：[位置] + [血量] + [攻击力]
- 小朋友 B 带了：[位置] + [血量]
- 小朋友 C 带了：[位置] + [攻击力]

**Archetype（原型） = "组件组合的类型"**

上面有 3 种不同的 Archetype：
1. {位置, 血量, 攻击力}
2. {位置, 血量}
3. {位置, 攻击力}

### 4.2 为什么这么设计？（性能秘诀）

```
传统做法（面向对象）：
玩家对象 = {位置, 血量, 攻击力, 技能, 装备, ...}
散落在内存各处，CPU 读取时跳来跳去，慢！

Archetype 做法：
Archetype{位置,血量,攻击力} 的内存：
  位置: [P1][P2][P3][P4]...  <-- 连续排列
  血量: [H1][H2][H3][H4]...  <-- 连续排列
  攻击: [A1][A2][A3][A4]...  <-- 连续排列
CPU 一次读一大片，飞快！这叫"缓存友好"
```

### 4.3 核心类图

```
World（世界，总管理器）
  ├── Entity（实体，只有一个 ulong ID）
  │     内部结构: [版本号|Archetype编号|在Archetype中的索引]
  │
  ├── Archetype[]（原型数组）
  │     每个 Archetype:
  │       ├── ComponentTypes（含哪些组件类型）
  │       ├── ComponentArrays（每种组件的数据数组）
  │       └── Entities（此原型下的所有实体ID）
  │
  ├── Systems[]（系统数组）
  │     每个 System:
  │       ├── Query（我关心哪些组件）
  │       └── Update(World, deltaTime)（每帧执行一次）
  │
  └── EntityManager（创建/销毁实体，增删组件）

Component（组件，纯数据 struct，不能有方法）
  ├─ PositionComponent { X, Y, Z }
  ├─ HealthComponent { CurrentHp, MaxHp }
  ├─ AttackComponent { AttackPower }
  └─ ...（随游戏扩展）
```

### 4.4 系统（System）怎么工作

```
// 例子：战斗系统想处理"有血量+有攻击力"的实体
// 它会这样做：
foreach (var archetype in world.Archetypes)
{
    if (archetype 既有 HealthComponent 又有 AttackComponent)
    {
        // 直接拿到连续的数组，飞快地批量处理！
        var healths = archetype.GetArray<HealthComponent>();
        var attacks = archetype.GetArray<AttackComponent>();
        for (int i = 0; i < archetype.Count; i++)
        {
            // 处理第 i 个实体的战斗逻辑
        }
    }
}
```

## 第五章：场景与 AOI 设计

### 5.1 什么是 AOI（Area of Interest）？

想象你在大广场上：
- 你需要知道：你身边 50 米内有谁、有什么
- 你不需要知道：广场另一边 500 米外的陌生人在干嘛
- **AOI = "我关心的范围"**

### 5.2 九宫格算法（最经典的 AOI）

把地图切成一个个格子（比如每格 50 米 × 50 米）：

```
┌───┬───┬───┐
│   │   │   │   ← 玩家在中间格子
├───┼───┼───┤
│   │ ★ │   │   ← 只需要知道"中间格子 + 周围8格"的信息
├───┼───┼───┤
│   │   │   │   ← 其他格子完全不关心
└───┴───┴───┘
```

**优点**：实现简单、查询快  
**适用**：玩家分布相对均匀的大多数MMORPG场景
